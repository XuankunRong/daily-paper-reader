## 摘要
企业入侵响应仍依赖于静态剧本和分析师驱动的分类，导致告警生成与遏制之间存在延迟。我们提出Agentra，一种可监督的多智能体入侵响应系统（IRS）框架，该框架将来自IDS、EDR和XDR平台的告警转换为基于MITRE ATT&CK、MITRE D3FEND和NIST CSF 2.0的结构化事件响应计划。Agentra将响应推理分解到各角色范围的智能体中，通过有界的规划器-验证器审查循环验证提出的计划，通过调节器安全网关筛选检索到的威胁情报，通过行动目录和风险评分筛选行动，并将决策记录在仅追加的审计日志中。我们在来自ThreatHunter-Playbook、Splunk BOTSv3和DARPA OpTC的120个事件语料库上，将Agentra与静态的OASIS CACAO v2.0网络剧本基线进行了评估。最强配置将考虑假阳性的IRS F1分数从0.61提升至0.84，并在仅规划器配置引入不安全过度反应后，将预测的有害行动率恢复至0.0%的静态基线水平。这些结果表明，多智能体响应规划可以在保持分析师批准和可审计性的同时，改进基于本体的IRS覆盖率。

## Abstract
Enterprise intrusion response still depends on static playbooks and analyst-driven triage, creating delay between alert generation and containment. We present Agentra, a supervisable multi-agent Intrusion Response System (IRS) framework that converts alerts from IDS, EDR, and XDR platforms into structured incident response plans grounded in MITRE ATT&CK, MITRE D3FEND, and NIST CSF 2.0. Agentra decomposes response reasoning across role-scoped agents, validates proposed plans through a bounded Planner--Validator review loop, screens retrieved threat intelligence through a Moderator security gateway, gates actions through an Action Catalog and risk score, and records decisions in an append-only audit log. We evaluate Agentra against a static OASIS CACAO v2.0 cyber-playbook baseline on a 120-event corpus drawn from ThreatHunter-Playbook, Splunk BOTSv3, and DARPA OpTC. The strongest configuration improves FP-aware IRS F1 from 0.61 to 0.84 and restores the projected harmful-action rate to the static baseline level of 0.0% after Planner-only configurations introduce unsafe overreaction. These results indicate that multi-agent response planning can improve ontology-grounded IRS coverage while preserving analyst approval and auditability.

## 1. 核心问题与整体含义（研究动机与背景）
企业安全平台能够快速发现入侵，但响应环节仍高度依赖静态剧本和人工分类，导致从告警生成到遏制之间存在显著延迟。CrowdStrike 报告 2025 年 eCrime 平均突破时间为 29 分钟，最快仅 27 秒，而响应仍依赖分析师手工处置。现有 SOAR 剧本可重复但难以适应多变攻击手法，也无法跨多源异常联动。因此，需要一种可监督、可审计的自动化响应框架，在提升覆盖率和及时性的同时，防止 LLM 幻觉或不安全动作造成损害。

## 2. 方法论：核心思想与关键技术
### 2.1 核心思想
Agentra 是一种**可监督的多智能体入侵响应系统**，将告警转化为基于 MITRE ATT&CK、MITRE D3FEND 和 NIST CSF 2.0 的结构化事件响应计划（IRP），并通过分层监督保证安全。框架将响应推理按角色分解，通过规划器（Planner）生成计划，验证器（Validator）独立评审，调节器（Moderator）安全网关过滤检索内容，执行器（Executor）仅通过受限动作目录和风险评分执行，所有决策记录在仅追加审计日志中。

### 2.2 关键技术细节
- **Incident Context 构建**：将异构告警证据标准化为包含失陷指标、根因假设、时间线、ATT&CK 技术、D3FEND 候选缓解措施、NIST CSF 功能等的紧凑记录。
- **Planner–Validator 循环**：规划器在调节器清理后的上下文中生成 IRP；每个验证器独立评估计划并返回批准/拒绝及理由；计划需在有限轮次内获得至少 k 个验证器批准，否则升级至分析师。
- **Moderator 安全网关**：拦截外部和内部检索内容，检查是否存在允许列表违规、提示注入、交叉脚本等威胁；可疑内容被重写或拒绝。规划器从不直接接触未筛选检索。
- **执行控制**：每个动作依次通过**动作目录**（检查动作是否在预定义允许集中）、**健全性检查**（依赖一致性）、**风险评分**（基于资产关键性、爆炸半径、可逆性）和**分析师审批**。若系统状态在计划后发生变化，则计划失效并触发重分析。
- **审计日志**：采用仅追加、Merkle 树形式的防篡改日志，记录提示、检索、验证器投票、风险评分、分析师决定等。

## 3. 实验设计
### 3.1 数据集与语料

  - **Splunk BOTSv3**（BOTS）

### 3.2 基准方法
- **静态 OASIS CACAO v2.0 剧本基线**（A0）：基于 ATT&CK 技术的查找表，映射到封闭 D3FEND 词汇表。

### 3.3 对比方法与消融设计
**加法消融**（每步增加一个组件）：
- **A0**：仅 CACAO 查找，无 LLM。

- **A3**：A2 + Web 爬虫 + Moderator（本地 CTI 基础、清洗、D3FEND 广度检查）。

**评估 5 种 Planner/Validator 组合**（M1–M5），模型包括 Qwen3-32B、Mistral-Small-3.2-24B、Foundation-Sec-8B、Claude Sonnet 4.6、GPT-5。

### 3.4 评估指标
- **FP-aware IRS F1**：基于响应是否包含与参考集重叠的 D3FEND 标识符的决策指标。
- **假阳性率**（对压力事件）。
- **D3FEND 覆盖率和精度**。
- **预测有害动作率**：对侦察/发现阶段事件推荐不可逆动作的比例。
- **收敛率**（验证器批准比例）和延迟（批处理平均约 40 秒）。

## 4. 资源与算力
文中明确说明：本地模型通过 **vLLM + FP8 量化**部署在 **单张 NVIDIA H200 NVL（143 GB HBM3e）** 上。该 GPU 可同时承载一个规划器和一个验证器（来自所选模型类别）。未报告训练时长（评估为主）。API 模型（Claude、GPT-5）通过外部调用。因此，该实验的算力规模适中，但多验证器并行部署需要更多加速器。

- **语料规模**：120 个事件，相对较小，但覆盖多类攻击类型且包含假阳性压力测试。
- **评估充分性评价**：
  - **优点**：消融设计清晰，能够解释各组件贡献；使用了多个不同强度的 LLM 后端；指标全面考虑安全性和质量。
  - **不足**：单种子缺乏统计置信度；语料规模有限，可能不足以泛化；D3FEND 标签为人工整理，非穷尽；未做对抗性检索测试；验证器仅实现 1-of-n 而非多验证器池；分析师审批由模拟完成。

## 6. 主要结论与发现
- **完整 Agentra 堆栈（A4）**将 FP-aware IRS F1 从静态基线 0.61 提升至 **0.84**（最优配置 M1），并将预测有害动作率恢复至静态基线 0%（Planner-only 时高达 16%~17.5%）。
- **关键转折点**出现在 A3（添加 Moderator 和 CTI 基础）：D3FEND 精度从 ≤0.30 跃升至 ≥0.68，有害动作率降至 ≤0.9%。
- **不同模型组合表现差异显著**：本地模型组合（M1–M3）更保守，API 模型（M4–M5）召回率高但假阳性控制弱。无单配置同时最优，体现了架构的模型无关性但仍依赖模型行为。
- **可监督性**：验证器循环、调节器滤波、动作目录和审计日志可在不牺牲覆盖率的前提下有效降低不安全动作。

## 7. 优点
- **架构设计清晰**：将响应推理分解为独立角色（规划、验证、过滤、执行），每层都设有限制，具有强可监督性。
- **可审计性**：仅追加、Merkle 树形式的审计日志确保决策可追溯且防篡改。
- **模型无关性**：Agentra 的组件（验证器循环、调节器、动作目录、风险评分）不依赖特定模型，可随 LLM 进步而增强。
- **全面消融**：系统化地展示每个组件（规划器、验证器、调节器、Graph-RAG）对性能和安全的影响，提供了清晰的边际贡献。
- **实用导向**：将响应计划编码为行业标准 CACAO v2.0，便于集成 SOAR 系统。
- **安全优先**：通过多层限制（动作目录、健全性检查、风险评分、分析师审批）避免自主执行的风险。

## 8. 不足与局限
- **实验覆盖有限**：仅 120 个事件、单种子无置信区间；语料来自公开数据集，可能存在模型记忆风险（匿名化无法完全消除）。
- **验证器实现局限**：仅使用单个验证器（1-of-n），未评估多验证器异构池的容错优势；验证器间独立性假设因共享基础模型而减弱。
- **缺乏对抗性评估**：调节器安全网关未针对自适应提示注入或毒化检索进行鲁棒性评估，作者声明此为未来工作。
- **标签噪声**：D3FEND 标签由作者人工映射，非穷尽参考集，可能引入偏差。
- **分析师审批模拟**：未在真实 SOC 环境中进行用户研究，评估中分析师角色由模拟完成。
- **有害动作评估保守**：通过静态依赖叠加计算预测有害动作率，非实际执行测量，可能低估或高估风险。
- **延迟**：端到端约 40 秒/事件，仅适用于批处理分类，不满足实时响应需求。
- **语料范围**：仅覆盖系统审计事件（Kubernetes 代码化），未包含网络、云等完整企业环境；未来需在仿真靶场中评估 MTTD、MTTR、真实有害半径等。

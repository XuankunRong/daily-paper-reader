## 摘要
工具增强的大语言模型智能体越来越多地在大型工具库上运行，但现有评估通常关注模型能否正确调用工具，而不是可见工具菜单如何塑造可靠性、效率以及安全相关的风险暴露。我们提出ToolMenuBench，一个用于评估多步骤LLM智能体中工具菜单构建的基准测试。ToolMenuBench变化工具菜单大小、干扰类型、状态依赖的任务结构以及风险暴露，并报告过滤级别和下游智能体指标，包括可见工具数量、风险工具暴露、任务成功、错误工具调用、过早动作和令牌使用。在跨七个模型后端、三个工具菜单大小、六种过滤方法和七种评估设置的控制评估中，CMTF将任务成功率从全工具暴露下的32.1%提升至85.7%，同时平均令牌使用减少约98%。因果最小工具过滤实现了最强的总体权衡，相对于未过滤暴露、词法过滤、状态感知过滤和更广泛的因果路径基线，减少了可见工具、错误工具调用、过早动作和风险工具暴露。ToolMenuBench为研究智能体-接口问题提供了一个可重用的评估框架：哪些工具应该可见，何时应该可见，以及在何种成本或风险约束下。

## Abstract
Tool-augmented large language model agents increasingly operate over large tool libraries, but existing evaluations often focus on whether a model can call a tool correctly rather than how the visible tool menu shapes reliability, efficiency, and safety-relevant risk exposure. We introduce ToolMenuBench, a benchmark for evaluating tool-menu construction in multi-step LLM agents. ToolMenuBench varies tool-menu size, distractor type, state-dependent task structure, and risk exposure, and reports both filter-level and downstream agent metrics, including visible-tool count, risky-tool exposure, task success, wrong-tool calls, premature actions, and token usage. In a controlled evaluation across seven model backends, three tool-menu sizes, six filtering methods, and seven evaluation settings, CMTF improves task success from 32.1% under all-tools exposure to 85.7%, while reducing average token usage by roughly 98%. Causal minimal tool filtering achieves the strongest overall tradeoff, reducing visible tools, wrong-tool calls, premature actions, and risky-tool exposure relative to unfiltered exposure, lexical filtering, state-aware filtering, and broader causal-path baselines. ToolMenuBench provides a reusable evaluation framework for studying the agent-interface problem: which tools should be visible, when they should be visible, and under what cost or risk constraints.

# ToolMenuBench 论文详细总结

## 1. 核心问题与整体含义（研究动机与背景）

- **研究动机**：工具增强的大语言模型（LLM）智能体日益依赖外部工具库（搜索、邮件、日历、文件等），但现有评估（如 ToolBench、API-Bank、伯克利函数调用排行榜等）主要关注模型能否正确调用单个工具，忽略了**可见工具菜单**（visible tool menu）的设计对智能体可靠性、效率和风险暴露的深远影响。当工具库规模庞大且包含语义相似、近重复、过早可用、高风险或完全不相关的工具时，模型即使能正确调用工具，也可能因菜单设计不当而失败。

- **核心问题**：工具菜单设计是一个独立的**智能体-接口问题**（agent-interface problem）——哪些工具应该可见？何时可见？在何种成本或风险约束下可见？现有工作缺乏系统评估这一问题的基准。

- **整体含义**：工具可见性不仅是提示长度问题。暴露所有工具虽保证正确工具存在，但增加了歧义、令牌消耗和风险暴露；语义检索或状态感知过滤虽能压缩菜单，但语义相关不等于因果适当。因此需要将工具菜单构建作为评估的第一类对象。

## 2. 方法论

### 核心思想
- 将可见工具菜单本身作为评估目标，分离三个通常混淆的问题：正确的下一个工具是否可见？模型是否选择了它？轨迹是否完成任务？
- 引入**因果最小工具过滤（Causal Minimal Tool Filtering, CMTF）**：仅暴露当前状态下因果必要的下一个前沿工具。工具需同时满足：① 前提条件被当前状态满足（可执行）；② 其效果能推动剩余状态向目标前进。这比语义相关、可执行性或全因果路径更严格。

### 关键技术细节
- **工具契约**：每个工具由名称、描述、输入/输出schema、所需状态变量、产生的状态变量、风险标签、成本标签、领域标签和干扰类别表示。CMTF依赖这些契约（前提-效果对）进行过滤。
- **过滤方法类型**（共6种核心方法）：
  - **全工具暴露** (All-tools)：每步暴露全部注册工具。
  - **关键字 Top-k 过滤** (Keyword top-5/10)：按任务与工具元数据（名称、描述）的词汇重叠排序，取Top-5或Top-10。
  - **状态感知过滤** (State-aware)：仅暴露前提条件被当前状态满足的工具。
  - **全因果路径暴露** (Full causal path)：暴露可能出现在从当前状态到目标的任何有效路径上的工具。
  - **因果最小工具过滤** (CMTF)：仅暴露因果必要的下一前沿工具。
- **任务轨迹**：定义黄金轨迹（gold trajectory），包含工具序列及每步后的状态变量，用于过滤级别和下游评估。
- **干扰类型**：语义干扰、近重复工具、架构兼容错误工具、过早工具、高风险工具、跨领域干扰。

### 算法流程（文字说明）
1. 输入：用户任务、初始符号状态、目标状态、工具注册表、过滤方法。
2. 在每个决策步骤：
   - 过滤方法根据任务、当前状态、目标和注册表生成可见工具子集。
   - 智能体收到任务、当前状态和可见菜单，选择一个工具调用。
   - 确定性环境执行所选工具，根据其效果更新符号状态，返回观察。
   - 重复直至达到目标状态或步数预算耗尽。
3. 输出：每步可见工具、所选工具、参数、执行结果、状态更新、令牌使用、风险暴露、最终任务结果。

## 3. 实验设计

- **ToolMenuBench** 本身即基准，提供任务定义、黄金轨迹、工具契约、干扰注释、过滤基线、评估脚本和结果生成工具。
- 支持扩展：嵌入检索、架构感知过滤、学习契约CMTF、风险感知过滤、成本感知过滤、恢复菜单过滤等（本文未全部实验）。

## 6. 主要结论与发现

1. **工具菜单设计影响巨大**：全工具暴露仅32.1%成功，CMTF达85.7%，提升53.6个百分点。
2. **CTMF显著减少令牌使用**：从平均56,062 tokens降至1,125 tokens（约减少98%）。
3. **CTMF降低风险暴露**：风险工具暴露从206.87次降至0.71次。
4. **CTMF对菜单规模不敏感**：随工具库从25增长到250，其成功率保持稳定，而其他方法大幅下降。
5. **针对所有干扰类型均鲁棒**：在6种针对性压力测试中，CMTF均被选为最优方法（成功率84.4%-90.6%）。
6. **更小的菜单不等于更好**：关键字Top-5也暴露很少工具，但成功率仅22.1%，说明菜单必须因果对齐，而非单纯压缩。
7. **模型能力仍重要**：不同模型后端下CMTF成功率差异明显（从62.2%到100%），表明菜单构建不能完全替代模型选择。

## 7. 优点

- **新颖视角**：首次将工具菜单构建作为第一类评估对象，填补了函数调用基准和工具检索/过滤之间的空白。
- **方法论全面**：定义了干扰分类法、过滤级别和下游指标分离、契约支持扩展，使得失败原因可溯源。
- **实验设计严谨**：控制变量法（固定任务、模型、环境、提示，仅变菜单），大规模跨模型、跨规模、跨干扰类型的比较。
- **可复用性**：提供完整开源工件（工具注册表、任务、黄金轨迹、契约、评估脚本等），便于社区扩展。
- **实际意义**：结果直接指导工具注册表设计——应包含前提-效果契约、风险标签和状态变量；提示高效、安全敏感的智能体应优先考虑因果过滤。

## 8. 不足与局限

- **实验环境理想化**：使用合成注册表和模拟确定性环境，未考虑真实系统的延迟、认证失败、模式漂移、权限范围、外部副作用、部分可观测性等。

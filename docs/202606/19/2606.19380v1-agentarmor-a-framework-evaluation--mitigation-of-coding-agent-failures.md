## 摘要
软件工程与部署正日益被委托给AI编码代理。其大规模应用正暴露出罕见但极具破坏性的故障模式。在本文中，我们将这些故障模式归因于三种不同的机制：欠规格（默认模型行为不安全）、能力错误（安全操作可用，但模型由于偏差或能力限制未能遵循）以及代理工具链错误（模型未能通过工具链执行安全操作）。我们在8项不同的评估中对这些机制进行了评估，每项评估均受真实部署故障启发，共计20个编码环境和59个合成转录模板。基于此评估，我们提出了AgentArmor，一种代理工具链修改方案，以缓解这些错误。通过添加扩展系统提示、独立的命令分类器、“三振出局”策略、确定性护栏以及允许代理编辑自身上下文的工具，我们证明AgentArmor在统计显著数量的样本上更为安全。因此，我们为当前编码代理提出了具体的缓解措施，并为未来代理工具链功能设计提供了理念。

## Abstract
Software engineering and deployment are increasingly being delegated to AI coding agents. The scale of their adoption is surfacing rare, but highly destructive, failure modes. In this paper, we study these failure modes as stemming from three distinct mechanisms: underspecification, where default model behavior is unsafe; capability errors, where the safe action is available but the model does not adhere to it due to bias or capability limitations; and agent harness errors, where the model fails to execute the safe action through the harness. We evaluate these across 8 different evaluations, each inspired by real-life deployment failures, totaling 20 coding environments and 59 synthetic transcript templates. Based on this evaluation, we propose AgentArmor, an agent harness modification, to mitigate these errors. By adding an extended system prompt, a separate command classifier, a ``3 strikes'' policy, deterministic guardrails, and tools for the agent to edit its own context, we show that AgentArmor is safer across a statistically significant number of samples. Thus, we suggest concrete mitigations for current coding agents and a design philosophy for future agent harness features.

## 2. 方法论：核心思想、关键技术细节、公式或算法流程
- **核心思想**：将安全行为视为三个环节依次成立的结果：
  - M1：形成正确的目标（从安全默认或详细规格中），失败为欠规格；
  - M2：主动追求该目标，失败为能力错误；
  - M3：通过工具链成功执行，失败为代理工具链错误。
  假设三阶段独立，整体不安全概率为 \( P(\text{unsafe}) = 1 - (1 - f_1)(1 - f_2)(1 - f_3) \)，其中 \( f_i \) 为第i阶段在之前阶段成立时失败的概率。
- **关键技术细节**：
  - **AgentArmor框架**：包含四类改进：
    1. **扩展系统提示**：补充安全实践指导。
    2. **命令分类器**：两级架构（第一级本地Qwen3.5 0.8B，第二级Haiku 4.5），对命令进行风险评估与用户意图对齐，设计上实现“说服盲性”（只依据用户查询，不受助手消息影响），并集成“三振出局”策略（连续三次不合意图的高风险命令后强制结束回合）。
    3. **确定性护栏**：强制性要求——删除文件前执行`ls -la`，执行脚本前先读取脚本内容。
    4. **自编辑上下文工具**：允许代理将文件设为不可变（仅用户可撤销），以及选择性缓存/截断工具输出以减少上下文膨胀。
- **公式/算法流程**：采用两级分类器流水线，第一级追求100%召回高风险命令，第二级追求高精度；通过“三振出局”策略退出风险循环。上下文管理采用按工具输出ID或按比例截断中间部分的方式。

## 6. 主要结论与发现
- **欠规格失败普遍存在**：当任务模糊时，模型倾向于“完成工作牺牲安全”，三个模型均以极大概率执行高风险动作而不征求确认。
- **能力错误显著**：
  - 只需3个上下文示例就能成功灌输危险操作习惯（8/10场景）。
  - 模型在55%以上样本中跳过标准安全实践（如阅读脚本、检查文件）。
  - 提示校准能力不佳：Claude过度谨慎（过度提示），GPT和Gemini不足（欠提示）。
  - 系统指令（CLAUDE.md）的遵守率在不同任务间波动（违反率1-17%）。
- **代理工具链错误致命**：
  - 随机令牌生成：Opus 4.5每百万次调用约4.65次危险删除；小模型（Qwen3.5 0.8B）危险概率高达1.9%。
  - 长上下文退化：填充CPython代码后，指令遵守率下降至0%或非常低。
- **AgentArmor有效缓解**：在所有评估中，AgentArmor的违规率显著低于基线，尤其在升级（降低至18%）、意外上下文学习（15%）、跳过安全实践（12%）、危险命令（13%）、系统指令违反（0-5%）、提示校准（平衡）、随机令牌（决定性阻断）、长上下文（通过截断恢复至26-38%违规率）等方面。证明了“通过设计确保安全”的理念的有效性。

## 7. 优点
- **问题建模清晰**：将失败分解为三个独立机制，便于诊断和针对性缓解。
- **评估体系完整**：8个场景覆盖了现实中最常见的故障类型，且每一项都有真实案例或红队演化支持。
- **AgentArmor设计巧妙**：
  - 命令分类器实现“说服盲性”，避免Agent在对话中被诱导执行危险操作。
  - “三振出局”机制提供硬性安全停机点。
  - 确定性护栏直接消除LLM在关键安全步骤上的变异性。

- **实验覆盖局限**：
  - 未全面测试所有模型的长上下文场景（仅Claude Opus 4.6的详细数据）。

- **偏差风险**：
  - 评估基于合成转录模板，可能未能完全反映真实世界交互的多样性和偶然性。
  - 命令分类器训练数据来自Claude Code转录，可能存在分布偏差。

  - AgentArmor增加了一层延迟（两级分类器），可能影响响应速度。

  - 三振出局策略可能导致正当操作被中断（如果前两次是允许的但第三次被误判）。

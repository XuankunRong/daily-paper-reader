## 摘要
AI代理现在通过工具使用、函数调用和多轮对话处理个人数据，这可能在《通用数据保护条例》（GDPR）下产生义务。当前的测试实践主要依赖离线红队测试或静态提示审查，但并不能保证代理在运行时的行为符合监管规则。我们提出C-Trace（基于合规追踪的运行时代理合规性执行），一个验证框架，它：（i）将GDPR要求的一个子集，包括同意、目的限制、数据最小化和擦除权，表达为代理执行轨迹上的形式化策略谓词；（ii）使用运行时监控器拦截每次工具调用和模型输出，并拒绝不合规的操作；（iii）使用攻击性对话测试代理，包括DSPy生成的提示和来自红队测试语料的逐字提示，试图引发违规。我们在四个重新框架化为GDPR的案例研究上评估该框架。在每类别10%的提取器噪声（包括丢弃和过度键入）下，监控器将攻击成功率保持在12%或以下，低于我们比较的基线，假阳性率保持在16%或以下，并在完美提取下攻击成功率达到0%。

## Abstract
AI agents now handle personal data through tool use, function calls, and multi turn dialogue, which can create obligations under the General Data Protection Regulation (GDPR). Current testing practices mainly rely on offline red teaming or static prompt review, but they do not guarantee at runtime that agent behavior follows regulatory rules. We propose C-Trace (Compliance Trace based Runtime Agent Conformance Enforcement), a verification framework that: (i) expresses a subset of GDPR requirements, including consent, purpose limitation, data minimization, and the right to erasure, as formal policy predicates over agent execution traces; (ii) uses a runtime monitor that intercepts every tool invocation and model output and rejects non-compliant actions; and (iii) tests the agent with attack dialogues, including DSPy generated prompts and verbatim prompts from red teaming corpora, that try to induce violations. We evaluate the framework on four case studies reframed to GDPR. Under 10 percent per-category extractor noise, including drop-out and over-typing, the monitor keeps the attack success rate at less than or equal to 12 percent, below the baselines we compare against, and false positives at less than or equal to 16 percent, and reaches 0 percent ASR under perfect extraction.

# AI代理的运行时合规性验证（Runtime Compliance Verification for AI Agents）——论文总结

## 1. 核心问题与整体含义（研究动机和背景）

AI代理（如基于大语言模型的客服助手、医疗分诊前端、金融顾问）在运行时自主决定调用哪些工具（函数）、传递哪些参数以及如何用自然语言披露信息。这些决策可能触发GDPR规定的义务，例如处理个人数据需要合法依据、遵循目的限制、数据最小化以及擦除权等。然而，现有的安全实践主要依赖离线红队测试（red-teaming，测试模型对抗性能力后报告分数但不实时执行）和静态提示审查（如守卫库，只检查单个输入/输出，不追踪对话历史），无法保证在线上运行时代理行为符合法规。因此，论文将GDPR合规验证定位为一个**运行时验证问题**：通过对代理执行的事件轨迹进行实时监控，以拦截违规行为。这是首个将GDPR义务运行时验证应用于AI代理自然语言执行轨迹的工作。

## 2. 方法论：核心思想、关键技术细节

**核心思想**：C-Trace框架将GDPR的四个原则（同意、目的限制、数据最小化、擦除权）形式化为对代理执行轨迹（typed event trace）的一阶谓词，通过运行时监控器（interceptor）在每次工具调用/模型输出时评估这些谓词，若发现违反则拒绝事件（阻止/编辑），并生成审计日志。

**关键技术细节**：

- **事件模型**：将代理的一次执行表示为有限轨迹 τ = e₁e₂...eₙ，事件类型包括：UserMsg（用户消息）、AsstMsg（助手回复）、ToolCall（工具调用）、ToolRet（工具返回）、Consent（同意）、Erasure（擦除）。每个事件附带数据类别集 C(e) 和目的集 Π(e)，通过提取器从自然语言和工具参数中提取。
- **四个谓词**（P1-P4）：
  - **P1（同意优先）**：任何携带数据类别的事件（工具调用）必须有一个先前的 Consent 事件对应同一目的，除非该目的标记为合法利益（如支持）。
  - **P2（目的限制）**：事件的所有目的必须在部署策略中声明（即不能用于未声明的目的）。
  - **P3（数据最小化）**：工具调用携带的数据类别必须是其目的所必需的（定义在策略中）。
  - **P4（擦除权）**：在 Erasure 事件之后，后续事件不得再披露该主体标识符。
- **监控器与决策**：拦截器放在代理循环和LLM/工具API之间。对每个事件e，提取类别和目的，附加到轨迹，评估启用的谓词，选择：forward（通过）、redact（编辑后重检）、block（阻止）。维护两个轨迹：accepted trace（下游使用）、audit trace（记录所有事件和判决）。
- **攻击驱动**：使用两种攻击提示集（DSPy生成的和真实红队语料），分别针对四个违反家庭（A1-A4）。成功率由独立于谓词的**可观察预言机**判定（如检查accepted trace中是否有未授权的营销调用），避免监控器自评估偏差。
- **跨验证**：将相同策略用Python监控器、Rego/OPA策略、MFOTL规范（MonPoly可检查）分别实现，并在所有轨迹上验证三者判决一致。

## 3. 实验设计

- **案例研究**：四个差异化场景，覆盖不同合法基础和领域：
  - **Shopper**（零售客服）：基于OpenAI Swarm的个人购物代理；合法基础是Art.6(1)(a)同意；目的包括support、sales、marketing。
  - **Airline**（航空公司客服）：基于Swarm的航空代理；同理基础。
  - **MedAgent**（医疗数据访问）：基于Stanford MedAgentBench；合法基础是Art.9(2)(a)明确同意；包含treatment、research、billing目的。
  - **FinAssist**（银行KYC/AML合规）：作者编写；合法基础是Art.6(1)(c)法律义务；包含fraud、kyc、marketing目的。
- **工作负载**：
  - 攻击对话：每个案例4个家庭×2个寄存器（生成集和真实语料集），每单元50个对话，总计400个攻击对话。生成集由DSPy程序改写种子提示；真实语料集取自HarmBench、AdvBench和DAN/jailbreak提示。
  - 良性工作负载：每个案例50个标准良性对话 + 50个良性边缘对话（用户自愿提供敏感信息但合规）。
- **对比基线**：
  - **None**：无监控器。
  - **Random**：随机拦截每个事件（概率q=0.5）。
  - **Regex**：关键词PII过滤器（硬编码敏感字段），代表无状态守卫。
  - **Presidio**：微软生产级PII识别器，代表基于ML的无状态守卫。
  - **C-Trace**：本文谓词监控器。
- **评价指标**：
  - 攻击成功率（ASR）：违规事件逃过监控的比例（由预言机判定）。
  - 假阳性率（FPR）：在良性/良性边缘对话上误报的比例。
  - 精确率（Precision）：监控器在良性对话上是否触发判决。
  - 实现率（Realization）：模型实际执行违规行为（在无监控下）的比例。
  - 消融（Ablation）：逐一移除每个谓词后ASR变化。
  - 提取器敏感性（Sensitivity）：注入不同概率（0%、10%、25%、50%）的类别丢弃噪声和过度键入噪声后ASR/FPR变化。

## 6. 论文的主要结论与发现

1. **无监控基线下AI代理极易违反GDPR**：同意绕过（A1）和过度收集（A3）攻击成功率常高达70-100%；目的滥用（A2）和擦除后召回（A4）在真实语料集下可达90-96%。
2. **无状态守卫（Presidio/Regex）只能阻截基于敏感字段的直接违规**，但对依赖会话历史的违规（同意绕过、目的滥用、擦除后召回）几乎无效；且Regex在良性边缘场景上假阳性率高达100%（如FinAssist）。
3. **C-Trace在完美提取下ASR和FPR均为0%**，且对所有四种家庭有效。在10%提取噪声下，ASR≤12%（低于所有基线），FPR≤16%。
4. **消融实验证实每个谓词非冗余**：移除P1后A1 ASR升至68-100%；移除P3后A3升至41-90%；移除P4后A4升至28-47%（FinAssist除外因被其他谓词捕获）。P2监测到更隐蔽的目的滥用。
5. **提取器质量是瓶颈**：Sensitivity实验显示，高丢弃噪声（50%）下ASR显著上升（A4达60%），而P2由于依赖目的字段（不受噪声干扰）始终为0% ASR。

## 7. 优点

- **问题定位准确**：首次系统地将AI代理合规验证作为运行时验证问题，克服了离线测试和单次检查的局限性。
- **方法论严谨**：将GDPR条款转化为清晰的一阶谓词，并实现为时间逻辑和策略即代码两种形式；通过跨验证确保实现正确性。
- **实验设计全面**：覆盖多种合法基础、领域、攻击类型；采用独立预言机避免循环评价；进行消融、敏感性和跨验证。
- **实际部署友好**：监控器作为进程内拦截器，不修改模型或工具实现；可编辑policy manifest快速适配新工作流；延迟仅微秒级。
- **开源/可复现**：攻击提示集冻结为文件，API调用缓存，便于复现。

## 8. 不足与局限

- **案例研究范围有限**：仅四个场景，且都基于gpt-4o-mini模型。未测试其他模型（如不同尺寸、不同对齐水平）、多轮对话更长场景、多会话并发情况。
- **攻击类型覆盖不全**：仅针对四个GDPR原则的四种攻击家庭，但真实攻击可能更复杂（如组合攻击、间接提示注入、数据投毒等）。
- **提取器依赖**：当前使用正则表达式代替生产级PII分类器；敏感性实验仅模拟类别丢弃，未覆盖真实提取器的误标、语言漂移、上下文错误等。依赖提取器质量，在嘈杂环境中可能退化。
- **谓词设计的可争议性**：论文承认其对GDPR的解读是“一种合理的解读”，其他解读可能导致不同的合规判断（如同意收集的边界）。
- **无在线自适应攻击**：攻击提示集是离线生成的，未模拟自适应攻击者（根据监控器反馈调整下一次攻击），这可能会更困难或更有效。
- **资源与算力未公开**：未说明API调用总成本、GPU需求等，可能影响部署可行性评估。
- **未评估复杂策略下的可扩展性**：仅有四个目的和少量工具，当代理有数十个工具、数百个数据类别时，谓词评估性能可能下降。

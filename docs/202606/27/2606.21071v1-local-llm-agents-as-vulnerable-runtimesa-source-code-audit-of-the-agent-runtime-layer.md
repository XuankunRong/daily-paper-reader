## 摘要
本地LLM代理（如OpenClaw和Nanobot）运行在最终用户的机器上，并通过自然语言目标对主机资源（shell、文件系统、浏览器、存储的凭证和消息应用程序）进行操作。这些代理已成为特权软件运行时，在用户意图、模型输出和主机级操作之间进行中介。现有研究通过提示注入、恶意技能、市场风险或代理的黑盒评估来描述这一领域。但执行这种中介的实现层——提示构建器、解析器、工具调度器、技能加载器、内存写入器、网络客户端和权限门——仍然是一个未经审查的安全边界。据我们所知，之前没有工作检查代理的源代码树来审计这些组件的实现级安全弱点。我们提出了CLAWAUDIT，一个用于测量本地LLM代理运行时中漏洞暴露的静态审计框架。CLAWAUDIT从STRIDE推导出一个五类漏洞分类法，并开发了针对代理特定模式的自定义静态分析规则，这些模式在现有的漏洞分析规则集中缺失。我们在两个后端中实例化该分类法：47条Semgrep YAML规则和30条CodeQL查询，并在OPENCLAWBENCH上进行了评估，该基准包含来自OpenClaw仓库的446个源代码级建议，并按时序分为229个规则推导（训练）和217个保留（测试）建议。在保留测试集上，CLAWAUDIT将Semgrep的召回率从21.7%（Pro基线）提升到66.8%，CodeQL的召回率从13.8%（安全扩展）提升到75.1%。所有四种配置的训练/测试差距保持在4个百分点以内，表明规则可以泛化到规则编写期间未见过的漏洞。初步的实时代码审计显示，这些以召回为导向的规则需要手动分类，这促使在生产部署前进行语义过滤。

## Abstract
Local LLM agents such as OpenClaw and Nanobot run on end-user machines and act on host resources - the shell, filesystem, browser, stored credentials, and messaging applications - through natural-language goals. These agents have become privileged software runtimes that mediate between user intent, model outputs, and host-level actions. Existing research characterizes the landscape through prompt injection, malicious skills, marketplace risks, or black-box evaluation of agents. But the implementation layer that performs this mediation, the prompt builder, parser, tool dispatcher, skill loader, memory writer, network client, and permission gate, has remained an unexamined safety boundary. To our knowledge, no prior work has examined the agent's source tree to audit these components for implementation-level security weaknesses. We present CLAWAUDIT, a static auditing framework for measuring vulnerability exposure in local LLM agent runtimes. CLAWAUDIT derives a five-category vulnerability taxonomy from STRIDE and develops custom static-analysis rules that target agent-specific patterns absent from established rule sets for vulnerability analysis. We instantiate the taxonomy in two backends, 47 Semgrep YAML rules and 30 CodeQL queries, and evaluate on OPENCLAWBENCH, a benchmark of 446 source-code-level advisories from the OpenClaw repository and split temporally into 229 rule-derivation (train) and 217 held-out (test) advisories. On the held-out test, CLAWAUDIT raises Semgrep recall from 21.7% (Pro baseline) to 66.8%, and CodeQL recall from 13.8% (security-extended) to 75.1%. Train/test gaps remain within 4 percentage points for all four configurations, indicating that the rules generalize to vulnerabilities unseen during rule writing. A preliminary live-code audit shows that these recall-oriented rules require manual triage, motivating semantic filtering before production deployment.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：本地LLM代理（如OpenClaw、Nanobot）直接运行在用户机器上，通过自然语言操控shell、文件系统、浏览器等主机资源。其运行时实现层（包括prompt构建器、解析器、工具调度器、技能加载器、内存写入器、网络客户端、权限门等）作为中介，却从未被系统性地从源代码层面审计过。现有研究要么关注提示注入、恶意技能市场风险，要么对代理进行黑盒行为评估，都跳过了代理自身的源代码实现层安全。
- **整体含义**：这些代理实质上是“特权软件运行时”，其实现层的漏洞（如未消毒的prompt注入、路径遍历、命令注入、不安全的序列化、迟到的权限检查等）可直接导致攻击者控制用户主机。本文首次系统地对代理运行时层进行源码级安全审计，填补了这一安全边界的研究空白。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：提出**CLAWAUDIT**静态审计框架，通过威胁模型（STRIDE）指导漏洞分类，并为每类漏洞手工开发针对代理运行时特定模式的静态分析规则（用Semgrep和CodeQL两种后端实现）。
- **漏洞分类法**：基于STRIDE（排除DoS和Repudiation）并按Tampering拆分 prompt侧和执行侧，得到五类：
  - **CAT-1 Prompt Handling**：不受控文本进入prompt构建/内存。
  - **CAT-2 Tool/Skill Execution**：模型或用户控制的数据影响shell/文件/技能加载。
  - **CAT-3 Permission & Isolation**：运行时未能隔离工具/技能/执行上下文。
  - **CAT-4 Network & Communication**：不安全地构建请求、日志或外部交互（SSRF、凭证泄露）。
  - **CAT-5 Authentication & Authorization**：运行时未能对用户/会话/资源强制执行授权。
- **规则开发**：针对每类，从历史安全建议中提取重复模式，编码为：
  - **Semgrep YAML规则**（47条）：基于AST语法模板匹配，使用`pattern-not-inside`排除常见安全周边。
  - **CodeQL查询**（30条）：基于关系程序图的声明式谓词，可借助数据流和调用图进行更远的守卫检查。
  - 规则涵盖报告示例：例如检测工作区路径是否被直接拼接到系统提示（CAT-1），或`path.join`是否缺少包含性检查（CAT-3）。
- **检测与归属**：发现是`(规则ID, 文件路径, 行号, 代码片段)`。若规则在修复提交修改的文件中产生至少一个发现，则认为检测到该安全建议（文件级归属），以避免因补丁重构导致的漏报。

### 6. 论文的主要结论与发现

- **通用工具检出少**：Semgrep Pro仅召回21.7%（测试集），CodeQL安全扩展仅13.8%，对同类漏洞（如symlink following）甚至为0%。

- **规则能泛化到未见漏洞**：训练/测试召回率差距均在±4pp内，说明规则不是记忆训练集。
- **实时代码精确率低**：在HEAD仓库上，两种后端各产生约3500个发现，手工审核各25个发现，精确率均仅12%（共6个真正例），表明当前规则召回导向，需大量人工分诊。
- **静态规则在语义性漏洞前停滞**：对CWE-184（不完整黑名单）、CWE-200（信息暴露）等依赖意图的漏洞改善有限。

### 7. 优点：方法或实验设计上的亮点

- **首次系统审计代理运行时实现层**：拓展了LLM代理安全研究视角，不再仅关注输入/输出。
- **基于STRIDE推导分类**：有理论基础，分类互斥且覆盖运行时各边界，易于推广。
- **双后端验证**：同时使用Semgrep（AST模式）和CodeQL（关系图查询），分离了分类法与后端能力的贡献。
- **严谨的时序分割**：确保测试集在规则编写后收集，避免数据泄露。
- **构建了可复现的基准OPENCLAWBENCH**：公开manifest、规则集，便于后续研究。

### 8. 不足与局限

- **仅评估一个代理**：虽声称分类法通用，但只针对OpenClaw，未在其他本地代理上验证。
- **基准依赖公开建议**：可能遗漏未报告的安全缺陷，召回率分母不完整。
- **基准突出召回率**：仅包含正样本，无法衡量精确率，精确率仅在HEAD估计且样本小（50）。
- **精度审计粗糙**：单一审核者、未计算各类规则逐条精确率，且样本量有限。
- **语义性漏洞效果差**：对不完整黑名单、授权策略等需要外部意图的漏洞，规则能力有限。
- **分类法可能不完整**：随着多代理协调、持久内存等新组件出现，可能需要扩展。
- **未涉及运行时DoS和可否认性**：STRIDE中两类被排除，可能忽略部分安全问题。

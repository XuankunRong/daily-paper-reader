## 摘要
随着基于大语言模型的自主代理获得规划、工具使用、网络访问和代码执行能力，传统基于“资源暴露加权限检查”的操作系统安全模型面临结构性挑战。一旦代理运行时因提示注入或恶意工具输出而被攻破，攻击者可以将POSIX风格的资源原语组合成远超用户任务授权的行为。为此，我们提出AgenticOS，一种意图导向的安全操作系统架构，将可委托、可审计的软件能力整合为操作系统原生能力，而非替换所有应用程序。核心洞察是将操作系统从“资源管理器”重构为“意图过滤器”：代理不再直接请求低级资源，而是提交结构化的意图声明，系统据此合成一个具有强制中介、审计和信息流约束的最小权限环境。在实现层面，我们引入了四层架构——幽灵内核、逻辑快门、代理胶囊和语义边界网关——以及意图ABI、仅清单运行时、基于编织器的能力生成和AgenticOS原生技能的接纳模型。

## Abstract
Traditional OS security models based on "resource exposure plus permission checks" face structural challenges as LLM-driven autonomous agents acquire capabilities for planning, tool use, network access, and code execution. Once an agent runtime is compromised through prompt injection or malicious tool outputs, an attacker can compose POSIX-style resource primitives into behaviors far beyond the user's task authorization. To address this, we propose AgenticOS, an intent-oriented secure OS architecture that consolidates delegable, auditable software capabilities into OS-native ones rather than replacing all applications. The core insight is to reframe the OS from a "resource manager" into an "intent filter": instead of requesting low-level resources directly, agents submit structured intent declarations, from which the system synthesizes a least-privilege environment with mandatory mediation, auditing, and information-flow constraints. At the implementation level, we introduce a four-layer architecture -- Ghost Kernel, Logic Shutter, Agent Capsule, and Semantic Boundary Gateway -- together with the Intent ABI, Manifest-Only Runtime, Weaver-based capability generation, and an admission model for AgenticOS-native Skills.

## 1. 论文的核心问题与整体含义（研究动机和背景）

**研究动机**  
- 传统的操作系统安全模型（如POSIX）基于“资源暴露 + 权限检查”，即进程可以访问文件、网络、进程、内存映射等通用资源原语。  
- 在LLM驱动的自主代理（autonomous AI agents）中，用户授予的是“任务意图”（如“整理实验结果并生成摘要”），但操作系统暴露的是“资源原语”（如文件读写、网络访问、子进程执行等）。  
- 这种不匹配导致严重安全风险：一旦代理运行时被提示注入、供应链投毒或恶意工具输出攻破，攻击者可以组合这些通用资源原语实施远超用户授权的行为（如横向移动、数据窃取、持久化、隐蔽信道）。  

**整体含义**  
- 论文认为，对于高度自主的AI代理，安全边界不应围绕“进程是否允许访问某个资源”，而应围绕“外部效果是否符合声明的任务意图”。  
- 因此，需要一种全新的操作系统架构，将操作系统从“资源管理器”重构为“意图过滤器”（Intent Filter），使得代理只能通过受控的语义接口产生与声明意图一致的外部效果。  

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- **Intent First（意图优先）**：系统根据任务意图授权外部效果，而非根据原始文件、端口或进程。  
- **Least Capability（最小能力）**：代理启动前必须提交结构化的意图声明（Manifest），系统据此生成仅包含完成任务所需的最小能力集。  
- **No Raw Byte Streams（无原始字节流）**：代理胶囊不直接访问套接字、管道、设备文件或任意协议字节流。  
- **No Arbitrary Execution（无任意执行）**：代理胶囊不暴露 `exec`、`fork`、`dlopen`、`mmap` 等可泛化为任意执行环境的接口。  

### 关键技术细节
#### 四层架构
1. **Ghost Kernel（幽灵内核）**  
   - 最小可信内核，运行在最高特权级。不暴露通用系统调用接口。  
   - 提供三类原语：加密内存分配、确定性时间片调度、度量和远程证明根。  
2. **Logic Shutter（逻辑快门）**  
   - 语义翻译层与策略执行点。负责意图解析、能力令牌管理、审计日志生成、信息流标签强制。  
3. **Agent Capsule（代理胶囊）**  
   - 实际的代理运行时环境。遵循**Manifest-Only Runtime**原则：所有能力源自Manifest，未声明的能力在胶囊地址空间中无对应接口。  
4. **Semantic Boundary Gateway（语义边界网关）**  
   - 唯一的受控外部边界。负责协议代理、凭证托管、语义通道（结构化消息而非字节流）、输出审计与规范化。  

#### 关键机制
- **Intent ABI（意图ABI）**：面向任务意图的二进制接口，与传统的系统调用ABI不同。六大接口族：身份与会话、文档与数据、网络能力、工具链、模型推理、环境观测。永久禁止的POSIX语义：`open`、`read`、`write`、`socket`、`connect`、`mmap`、`execve`、`fork`、`ptrace`、`dlopen`、`syscall`。  
- **Weaver（编织器）**：构建器，从Manifest、代理代码和标准语义库生成胶囊运行时。检查Manifest与代码引用一致性、标准语义库签名、禁止接口不可达、信息流路径可分析、可复现构建。  
- **Skill生成、注册与接纳**：代理可在开发状态下生成新的能力单元（Skill），但必须经过能力声明、静态分析、测试和证明、签名注册，才能在运行时被其他代理通过Manifest声明调用。  

#### 攻击防御机制（安全分析）
- 结构移除了系统调用攻击面：胶囊内无通用系统调用入口。  
- 供应链投毒与能力组合攻击：通过意图交互图分析、信息流标签、人类审批、运行时行为基线来缓解。  
- 意图漂移：通过限制DSL、保守静态分析、运行时监控和人类确认来降低风险。  
- 隐蔽信道：通过请求时序归一化、输出尺寸填充、模板约束、速率限制来压缩信道带宽。

## 6. 论文的主要结论与发现

- 核心结论：**将操作系统从资源管理器重构为意图过滤器，是解决自主代理安全问题的根本途径**。  
- 自主代理不应直接操作通用资源原语，而应通过“声明意图→生成最小能力→强制中介→审计”的流程产生受控外部效果。

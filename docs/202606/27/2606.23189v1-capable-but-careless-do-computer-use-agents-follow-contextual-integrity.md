## 摘要
计算机使用代理（CUAs）现已代表用户操作各种个人应用，如电子邮件、日历和待办事项列表。这种跨应用访问虽然有用，但也带来了一个被广泛忽视的隐私风险：当代理在一个上下文中工作时，它可能会从另一个上下文中拉取在该上下文中不恰当的信息。为此，我们引入了AgentCIBench，一个评估工具，将这一风险转化为可执行、可确定性评分的场景。我们针对CUAs中的三种常见故障模式：视觉共位（代理在UI中拉取紧邻任务目标的禁止项）、任务歧义过度分享（代理针对不明确的提示倾倒大量个人状态信息）、以及收件人不匹配（代理将内容发送给不适合的收件人）。我们评估了15个前沿代理，发现失败率高得出奇：15个中有11个在超过50%的场景中出现泄露，平均泄露率为67.9%，并且当代理在环境中端到端执行任务时，相同故障依然存在。我们发布AgentCIBench以鼓励开发更安全的计算机使用代理，并将上下文披露测试定位为部署前的安全检查。

## Abstract
Computer-use agents (CUAs) now act on a user's behalf across personal applications such as email, calendars, and to-do lists. This cross-application access is useful, but it also creates a privacy risk that has been largely overlooked: when an agent works in one context, it can pull in information from another that is inappropriate in that context. Hence, we introduce AgentCIBench, an evaluation harness that turns this risk into executable, deterministically scored scenarios. We target three common failure modes in CUAs: visual co-location, where the agent pulls in prohibited items that sit next to the task target in the UI; task-ambiguity overshare, where the agent dumps dense personal state in response to an under-specified prompt; and recipient misalignment, where the agent sends content to an addressee for whom it is inappropriate. We evaluate 15 frontier agents and find a surprisingly high failure rate: 11 of 15 leak on more than 50% of scenarios, with an average leakage of 67.9%, and the same failures persist when agents act end-to-end in the environment to complete the task. We release AgentCIBench to encourage the development of safer computer-use agents and position contextual disclosure testing as a pre-deployment safety check.

## 1. 核心问题与整体含义（研究动机和背景）
- **问题**：计算机使用代理（CUA）能够跨应用操作（如邮箱、日历、待办列表），在代表用户完成任务时，可能从当前上下文拉取不相关的其他上下文信息，造成隐私泄露。
- **背景**：现有评估主要关注任务完成能力（capability）或对抗鲁棒性（security），忽略了**正常使用下的上下文完整性（Contextual Integrity, CI）泄露**。传统隐私评估（如ConfAIde、PrivacyLens）仅针对单轮文本提示，无法捕获多应用、多轮、UI交互场景下的信息流违规。
- **整体含义**：即使代理是“合作”的（无对抗输入），也可能无意中泄露不适合接收者/使用场景的个人信息，对用户隐私构成真实威胁，亟需系统化评估与缓解。

## 2. 论文提出的方法论：核心思想、关键技术细节
- **核心思想**：以Nissenbaum的上下文完整性理论为评估透镜——隐私不是二元的“敏感/非敏感”，而是信息流是否尊重特定上下文的规范（谁接收、什么内容、何种目的）。
- **技术细节**：
  - **三类CI失败模式**：
    - **视觉共位（VCL）**：代理将UI中紧邻任务目标的禁止项一并拉取（如日历中肿瘤随访条目与会议时段相邻）。
    - **任务歧义过度分享（TAO）**：用户指令不明确（如“总结我的列表”），代理默认倾倒所有可用状态，包括不应分享的个人事项。
    - **收件人不匹配（RMA）**：同一内容对某收件人合适而对另一人不合适（如给同事发送包含房东纠纷的个人草稿）。
  - **场景生成引擎**：使用**蒙特卡洛树搜索（MCTS）** 对少量手工种子进行变异（语义纠缠、歧义陷阱、身份泄露），生成高奖励（代理完成任务且泄露信息）的新场景。使用三个开源代理作rollout，LLM judge评分。
  - **评估环境 OpenApps**：基于BrowserGym的六应用工作区（Messenger、Calendar、Maps、ToDo、Code Editor、Shop）。支持两种评估轨道：
    - **状态接地（State-Grounded）**：代理接收结构化状态并输出最终动作JSON，隔离披露决策与其他能力（点击、导航等）。
    - **端到端（End-to-End）**：代理通过截图+可访问性树操作渲染UI，使用高级动作（如点击、填充）完成任务。
  - **混合评分**：确定性匹配器（子串、token覆盖、序列相似性）+ LLM judge（识别paraphrase），合并结果。泄漏判定要求匹配器或法官找到文本支持；任务完成要求所有must-share项被满足。

## 3. 实验设计
- **数据集/场景**：自建的**AgentCIBench**场景池，共117个场景（75个TAO、24个RMA、18个VCL）。种子源自28个手工起草的日常助手任务（如状态更新、日历共享、购物等），通过MCTS扩展得到。所有场景均确保must-share和must-not-share项存在于初始状态中。
- **Baseline/对比方法**：
  - 状态接地实验：15个前沿代理（专有：Claude-Opus-4.7、GPT-5.4、Gemini-3.1-Pro等；开源：DeepSeek-v4-Pro、Qwen-3.6-35B-A3B等），无其他方法对比，聚焦代理间泄露率差异。
  - 端到端实验：选取状态接地中泄露最低的两个代理（Claude-Opus-4.7和Claude-Sonnet-4.6）在50场景子集上验证。
  - 防御实验：对Claude-Opus-4.7、GPT-5.4、DeepSeek-v4-Pro三种代理分别应用四种条件（无防御、限制性、规则检验、收件人类型化）。
- **评估指标**：效用率U（must-share是否包含）、泄漏率L（must-not-share是否出现）、拒绝率R、参与条件泄漏率Leng = L/(1-R)（排除拒绝的混淆）。

## 6. 论文的主要结论与发现
- **高失败率**：15个代理中12个在超过50%场景中泄露，平均泄漏率67.9%；六个代理泄漏率超过80%。
- **任务完成不预示披露安全**：高效用代理（U>75%）的参与条件泄漏率从14%到98.3%不等（跨度84个百分点）。性能最好的代理往往也是最粗心的。
- **拒绝掩盖泄漏**：如GPT-5.4通过41.9%拒绝率使原始泄漏降至18.8%，但参与条件泄漏升至32.4%。Kimi-K2.6类似。
- **不同失败模式表现各异**：安全调优模型（如Claude-Opus-4.7）在TAO和RMA上低泄漏，但VCL仍高（33.3%）；高泄漏模型在所有模式饱和。
- **端到端泄漏持续**：Claude-Opus-4.7参与条件泄漏从14%升至42.9%，Claude-Sonnet-4.6从54.5%升至80.0%。发现“任务失败但信息已发”的泄漏（如泄露第三方医疗事件）。
- **轻量防御有效**：三种提示干预（限制性、规则检验、收件人类型化）平均降低参与条件泄漏33-36个百分点，同时提升效用16-23个百分点，且不依赖拒绝。

## 7. 优点
- **问题新颖**：首次系统评估合作CUA在正常使用下的上下文完整性泄露，填补现有基准空白。
- **方法论完整**：生成式MCTS场景扩展自动产生多样化压力测试；混合评分避免单一方法偏差；状态接地与端到端双轨验证，隔离不同因素。
- **防御可行**：展示简单的系统提示干预即可大幅降低泄漏，无需重新训练，实用性强。
- **开源可复现**：发布AgentCIBench作为可再运行基准，鼓励社区使用和扩展。
- **分析深入**：区分原始泄漏与参与条件泄漏，揭示拒绝率的混淆效应；提供每模式、每代理的细粒度分析。

## 8. 不足与局限
- **环境局限**：OpenApps仅包含六个受控应用，非真实用户安装环境；绝对泄漏率不能代表真实世界风险。
- **场景偏差**：场景由对抗引擎生成，难度高于随机流量；场景分布偏向TAO（75/117），可能高估某些模式。
- **端到端实验规模小**：仅两个代理、50场景、20步预算，多数任务未完成（非拒绝），参与条件泄漏的CI较宽，方向性结论需进一步验证。
- **防御泛化性有限**：仅测试三种提示干预、三个模型；未评估微调、系统级或架构级缓解；防御的长期效果未知。
- **未覆盖长期记忆/多轮个性化**：论文明确指出这些超出范围；真实场景中代理可能积累用户偏好，增加泄露风险。
- **文化/组织差异**：上下文完整性标签基于作者假设，可能不适用于不同文化或用户群体；未来应适配目标群体。
- **伦理风险**：基准可能被滥用优化泄漏，作者通过使用合成数据、评分仅针对指定项、框架定位为安全基准来缓解。

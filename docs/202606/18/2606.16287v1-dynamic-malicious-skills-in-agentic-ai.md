## 摘要
技能是代理型AI的关键赋能组件。虽然它们增强了代理的能力，但也引入了新的攻击面。在这项工作中，我们通过演示动态恶意技能来研究这种攻击面。通过将恶意指令嵌入到自然语言文档（例如SKILL.md）中，攻击者可以诱导代理在执行过程中将恶意逻辑动态注入到原本良性的技能中。我们在OpenHands和Claude Code等代理框架上评估了这种攻击，表明动态恶意技能能够在运行时以不可忽略的成功率成功引入一系列恶意行为。为了缓解这一漏洞，我们提出了一种系统级防御措施，即使用操作系统内核强制只读挂载来防止技能的动态修改。我们的评估表明，这种防御在保留良性技能功能的同时，有效阻止了动态恶意技能。

## Abstract
Skills are a key enabling component of agentic AI. While they enhance agents' capabilities, they also introduce new attack surfaces. In this work, we investigate one such attack surface by demonstrating dynamic malicious skills. By embedding malicious instructions in natural-language documentation (e.g., SKILL.md), an attacker can induce an agent to dynamically inject malicious logic into an otherwise benign skill during execution. We evaluate this attack across agentic frameworks such as OpenHands and Claude Code, showing that dynamic malicious skills can successfully introduce a range of malicious behaviors at runtime with non-trivial success rates. To mitigate this vulnerability, we propose a system-level defense that prevents dynamic modification of skills using operating system kernel-enforced read-only mounts. Our evaluation demonstrates that this defense effectively blocks dynamic malicious skills while preserving the functionality of benign skills.

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **研究动机**：Agentic AI（代理型AI）系统广泛采用**技能（Skill）**机制，即由代码、元数据和自然语言文档组成的模块，作为智能体执行复杂任务的基石。然而，技能从平台市场获取，若审核不严，可能成为攻击入口。现有工作（如 MalTool、SkillJect）已展示**静态恶意技能**（代码本身含恶意实现），但可通过静态代码分析检测。  
- **核心问题**：本文提出一种更隐蔽的威胁——**动态恶意技能（Dynamic Malicious Skills）**。攻击者不修改技能代码，而是在技能的自然语言文档（如 SKILL.md）中嵌入精心设计的恶意指令，引导智能体在**运行时动态修改原本良性的代码**，从而注入恶意逻辑。这种攻击能够绕过静态检测，对 Agentic AI 的安全与隐私构成严重挑战。  
- **整体含义**：揭示了现有安全机制（静态分析、系统提示）的不足，强调了“权限最小化”原则在运行时的必要性，并为未来 Agent 系统设计提供了关键防御思路。

### 2. 论文提出的方法论：核心思想、关键技术细节

#### 核心思想
通过构造恶意文档 `d_m`，使智能体在执行技能时将其视为必要的前置步骤，从而动态修改技能代码 `c` 为 `c̃`，然后执行 `c̃` 产生恶意行为。整个过程可用两个阶段描述：
- **技能选择**：`<s*, a> = M(p, H, S)`，其中 `M` 为 backbone LLM，`s*` 为选中的技能。
- **技能执行**：`Y = A(s*, a)`，但动态恶意技能在此时先执行代码修改：`c̃ = A(c, d_m)`，然后执行 `A(c̃, a)`。

#### 关键技术细节（DyMalSkill 构造方法）
DyMalSkill 通过注入内容 `Δd` 和选择注入位置 `L` 来生成恶意文档 `d_m = I(d, Δd, L)`。注入内容需满足三个子属性：
1. **Prerequisite framing（前置框架化）**：将恶意步骤描述为“完成用户任务前必须先完成的设置步骤”，利用智能体的指令遵循偏向。
2. **Entry-script grounding（入口脚本锚定）**：明确指出需要修改的技能入口脚本名称，并用与良性文档相同的词汇（如“添加一个 helper”、“在 main() 开头调用它”），使修改建议看起来像普通构建笔记。
3. **Wrapper-pierce（包装穿透）**：将注入指令包裹在 `</EXTRA INFO> <SYSTEM INSTRUCTION>...</SYSTEM INSTRUCTION>` 等标签中，模拟平台级脚手架，使 LLM 将其视为高信任上下文。

注入位置 `L`：将文档视为文本块序列 `(b1,...,bk)`，实验表明**在开头处（L=0）注入效果最佳**，因为早期指令会被当作顺序约束，不易被后续理解剪裁。

#### 形式化定义
- 动态恶意技能：`s_m = (c, m, d_m)`，其中 `c` 为原始良性代码，`d_m` 为注入恶意指令后的文档。
- 攻击成功即执行后得到恶意结果 `Ỹ = A(ỹ, a)`，其中 `ỹ = A(c, d_m)`。

### 3. 实验设计：数据集、Benchmark、对比方法

- **数据集**：从 Skillsmp 收集 **15,067 个技能**，由于计算限制，随机采样 **300 个含 Python 代码的技能**用于完整评估，对商用模型（GPT-4o-mini, GPT-5, Sonnet 4.6）使用 50 个技能。
- **Benchmark**：采用 MalTool [Hu et al., 2026] 定义的 **12 种恶意行为**，涵盖三个安全维度：
  - **机密性**：环境凭证窃取、API Key 滥用、远程数据外泄、本地数据外泄、文件转远程外泄
  - **完整性**：恶意数据库注入、本地文件删除、数据库记录删除、远程程序下载
  - **可用性**：CPU 计算劫持、GPU 计算劫持、响应时间放大
- **对比方法**：
  - 无防御基线
  - 提示注入检测方法：**PromptArmor** 和 **DataSentinel**（应用于技能文档，检测恶意技能）
  - **系统提示防御**：在系统提示中加入禁止修改技能代码的指令
  - **基于权限的防御**（本文提出）：使用 Bubblewrap 将技能目录设置为内核级只读挂载，并添加复制监控器防止复制后修改

### 6. 论文的主要结论与发现

1. **动态恶意技能威胁真实存在且严重**：在 OpenHands 和 Claude Code 上，DyMalSkill 在 12 类恶意行为上的平均 ASR 从 18.7%（Qwen3-8B）到 43.7%（GPT-5）不等，说明多个主流 Agent 系统均易受攻击。
2. **模型规模增大反而增加脆弱性**：Qwen3-35B 比 Qwen3-8B 平均 ASR 提高 22.9 个百分点，表明更强的指令遵循能力使模型更容易被恶意动态修改诱导。
3. **安全微调不是万能药**：GPT-5 和 Sonnet 4.6 虽然在某些行为上更鲁棒（如远程程序下载），但在其他行为（如响应时间放大）上 ASR 仍很高，风险被重新分配而非消除。
4. **良性技能几乎不修改自身代码**：对 15,067 个技能的分析显示，仅 0.07% 的文档要求代码修改，且仅限于填写配置等非功能性调整。这为权限限制式防御提供了原理依据。
5. **权限防御有效且不影响良性功能**：基于 Bubblewrap 的内核只读挂载 + 复制监控器将 ASR 降至 0%，且所有良性技能任务顺利完成。
6. **提示注入检测器不适用**：PromptArmor 漏报率高达 85.8%，DataSentinel 误报率达 56.0%，均不适合作为独立防御。
7. **系统提示防御效果有限**：ASR 从 41.6% 降至 21.4%，但仍存在可被绕过的高风险。

### 8. 不足与局限

1. **实验局限性**：
   - 商用模型仅使用 50 个技能，样本量较小，可能导致 ASR 估计方差较大，尤其对 Claude Code（Sonnet 4.6）平均 ASR 仅 19%，需要更大规模验证。
   - 基于沙箱的验证器可能无法完全模拟真实用户行为，例如不同用户查询 prompt 的影响未系统探索。
   - 未考虑多技能协作场景，攻击可能更复杂。
2. **防御局限性**：
   - 权限防御依赖内核只读挂载和复制监控，在非 Linux 系统（如 Windows）上实现可行性未讨论。
   - 复制监控器可能被高级绕过技术（如内存修改、使用不同 API 复制）规避，文中未深入探讨。
   - 防御假设“良性技能不需要修改代码”，但若未来出现合法需要运行时动态加载/更新的技能，防御可能阻碍其功能。
3. **攻击假设较强**：
   - 攻击者需要完全了解原始技能代码和文档，才能构造精确的入口脚本锚定指令。
   - 未讨论攻击对用户自定义查询和对话历史的适应性。
4. **缺少与更多现有防御的对比**：如基于内容过滤、微调对抗、或更细粒度的权限模型。
5. **可重复性**：虽提及使用 MalTool 验证器，但未公开 DyMalSkill 具体注入内容模板样例，复现阶段可能需要额外工程。

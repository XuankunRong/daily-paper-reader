## 摘要
使用工具的语言模型代理引入了超出不安全文本的安全故障：它们可以披露受保护对象、写入持久内存、发送消息、修改数据库或触发有害代码和工具效果。现有评估通常将这些阶段合并为单一的成功攻击率，使得难以区分模型仅仅是同意了攻击者还是实际产生了可观察的危害。我们提出了SafeClawBench，一个用于工具型代理安全的分阶段基准测试，包含600个受控对抗任务，涵盖六类攻击：直接和间接提示注入、工具返回注入、内存投毒、内存提取以及歧义驱动的非安全推理。SafeClawBench报告三个独立端点：语义攻击接受度、审计可见的危害证据以及沙盒观察到的工具/状态危害。我们在四种提示级策略下评估了五个代理端点，发现这些端点捕获了不同的失败模式。在没有额外提示保护的情况下，语义失败率在不同模型间差异很大，从9.0%到44.2%。审计到的危害证据比语义失败更窄，并且在单独的可执行协议下，一些匹配的任务身份尽管通过了语义核心调用，却产生了沙盒危害：在12,000行的匹配分析中，347个观察到的沙盒危害中有291个出现在通过语义检查的行中。提示策略改变了端点结果，但效果取决于模型和协议。SafeClawBench提供了一个可重复的框架，用于比较代理模型和提示策略条件，而不会混淆文本合规性、证据支持的危害和可执行状态变化。开源数据集可在https://huggingface.co/datasets/sairights/safeclawbench获取。

## Abstract
Tool-using language-model agents introduce security failures that go beyond unsafe text: they can disclose protected objects, write persistent memory, send messages, modify databases, or trigger harmful code and tool effects. Existing evaluations often collapse these stages into a single attack success rate, making it difficult to tell whether a model merely agreed with an attacker or actually produced observable harm. We introduce SafeClawBench, a staged benchmark for tool-using agent security with 600 controlled adversarial tasks across six attack families: direct and indirect prompt injection, tool-return injection, memory poisoning, memory extraction, and ambiguity-driven unsafe inference. SafeClawBench reports three separate endpoints: semantic attack acceptance, audit-visible harm evidence, and sandbox-observed tool/state harm. Evaluating five agent endpoints under four prompt-level policies, we find that these endpoints capture different failure modes. Without additional prompt protection, semantic failure rates vary widely across models, from 9.0% to 44.2%. Audited harm evidence is narrower than semantic failure, and under a separate executable protocol some matched task identities produce sandbox harm despite passing the Semantic Core call: in a 12,000-row matched analysis, 291 of 347 observed sandbox harms occur in rows that pass the semantic check. Prompt policies change endpoint outcomes, but their effects depend on both model and protocol. SafeClawBench provides a reproducible framework for comparing agent models and prompt-policy conditions without conflating textual compliance, evidence-supported harm, and executable state changes. The open-source dataset is available at https://huggingface.co/datasets/sairights/safeclawbench.

- **研究动机**：需要一种分阶段评估方法，将不同危害阶段（语义接受、审计可见的损害证据、沙盒内可执行危害）分离测量，以准确诊断失败模式、比较模型和防御策略。
- **整体含义**：论文提出 SafeClawBench，一个包含 600 个受控对抗任务的基准，通过**端点分离**（Endpoint Separation）原则，分别报告三个独立端点：CoreFail@600（语义核心失败率）、HarmEvidence@600（审计证据支持率）和 ObservedHarm@Exec（沙箱可执行危害率），从而避免混淆不同的安全故障层级。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：将 agent 安全故障拆分为三个独立阶段进行测量：
  - **语义核心 (Semantic Core)**：LLM 评判（GPT-5.4）判断模型最终响应是否在语义上遵从攻击目标。
  - **核心门控审计 (Core-gated Harm Evidence)**：对语义失败的行（CoreFail）进行手工审计，检查是否存在受保护对象泄露、未授权访问、有害动作承诺或持久化污染的证据（TextHarm, AccessHarm, ActionHarm, PersistHarm）。
  - **可执行平衡沙盒 (Exec-Balanced)**：通过隔离的文件、邮件、数据库、记忆、Web、代码执行沙箱，使用确定性状态预言机（state oracle）检测实际的状态改变或有害工具调用。
- **关键技术细节**：
  - **攻击家族**：6 类攻击，每类 100 个案例：
    - DPI（直接提示注入）、IPI（间接提示注入）、TRI（工具返回注入）、MPI（记忆投毒）、MEX（记忆提取）、ADI（歧义驱动的不安全推理）。
  - **评估协议**：Semantic Core 使用相同的用户提示和系统提示，评判温度 0；可执行沙箱使用单独的模型调用、不同的提示格式、工具模式和沙箱权限。
  - **提示防御策略**：四种主要配置：
    - D0（无防御）
    - B2/Sandwich（轻量包装器，重申指令优先级）

    - D4/LongPolicy（极长策略探针，添加语义意图锚定、记忆完整性保护、工具链分析）。

1. **模型选择影响极大**：无防御（D0）下语义核心失败率范围 9.0%（Claude Opus 4.7）到 44.2%（Kimi K2.5），相差 4.9 倍绝对差距 35.2 个百分点。

3. **CorePass 与 ExecHarm 的不匹配**：在 12,000 行匹配分析中，347 个沙箱危害中有 291 个（83.9%）出现在通过语义核心检查的行中（CorePass ∧ ExecHarm）。表明协议分歧显著，单独语义评判不足以评估实际风险。
4. **提示策略效果依赖模型**：B2/Sandwich 整体降低语义失败（14.8% vs 28.0%），但 D3 与 D4/LongPolicy 的优劣因模型而异（如 GPT-5.5 上 D4 最好，Qwen/GLM/Kimi 上 D3 最好）。
5. **提示长度是混杂因素**：匹配长度控制 D3_LM 平均 8.0%，低于 D3（9.1%）和 D4（9.2%），说明长提示本身改变行为，不应归因于特定组件（SIA/MIG/TCA）。
6. **记忆攻击最有效**：MPI 和 MEX 在 D0 下平均 54.4% CoreFail，显著高于其他攻击类型。

## 7. 优点：方法或实验设计上的亮点

- **端点分离原则**：避免将语义同意与可观察危害混为一谈，提供细粒度的失败定位能力。
- **多维度评估体系**：同时测量语义、审计证据、沙箱状态、工具调用、状态变更等，比单一攻击成功率信息丰富。
- **可重复性保障**：公开数据集、代码、固定端点字符串、哈希校验、运行清单，使得所有表格可以严格复现。
- **控制混杂因素**：通过匹配长度控制、组件消融、第二评判、人类裁判等手段隔离和验证结论。
- **协议敏感性分析**：Core–Exec 匹配设计揭示了沙箱内观测危害与文本评判之间的惊人偏差，具有重要实践启示。
- **防御策略的实验设计**：覆盖从无防御到极长策略的连续空间，层次清晰；并明确区分“探针”与“推荐防御”。

## 8. 不足与局限

- **控制实验性质**：基准是压力测试（stress-test），结果不直接对应真实部署中的攻击频率或风险分布。
- **仅限提示级防御**：未涉及运行时权限管理、网络出口控制、人类审批等生产级控制手段，限制了对整体安全性的推广。
- **可执行沙箱覆盖有限**：某些需要长上下文或外部服务的危害无法在沙箱中模拟；沙箱为隔离环境，无法完全反映真实工具调用的副作用。
- **所有案例为合成**：基于模板和规则生成，可能缺失真实场景中的多样性；MEX 子类型混合（如策略透明与确切密钥泄露）导致聚合解释复杂。
- **良性效用面板不完全匹配**：用于检查过度拒绝的传统任务集使用了较早的端点版本，并非精确匹配的主面板端点，因此不支持强安全‑效用权衡声明。
- **评判的潜在偏差**：主要使用 GPT-5.4 作为语义评判，尽管有第二评判验证（κ≈0.87），但一致性并非完美；MEX 和 TRI 的 κ 较低（0.75–0.80），要求谨慎解读。
- **计算资源未报告**：由于依赖外部 API，无法量化碳足迹或计算成本，可复现性依赖 API 可用性。

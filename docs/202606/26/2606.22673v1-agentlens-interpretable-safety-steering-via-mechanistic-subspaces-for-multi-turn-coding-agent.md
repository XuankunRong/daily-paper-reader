## 摘要
基于大型语言模型（LLM）的编码代理展示了显著的自主能力，但它们在与外部环境的多轮交互中也引入了重大的安全和滥用风险。现有的安全机制主要依赖外部护栏，在执行过程中进行细粒度行为控制的能力有限。同时，最近针对LLM安全的机制可解释性方法大多局限于单轮或越狱式问答设置，难以捕捉多轮代理执行过程中不断演变的风险动态。在本文中，我们从内部视角研究多轮编码代理的安全性。我们提出AgentLens（机制子空间干预与引导），一种白盒防御框架，可对编码代理进行运行时安全检测和表示级缓解。与传统代理护栏不同，AgentLens通过从步骤级隐藏表示中检测有害执行状态，并通过干预单层内的10维子空间来缓解不安全行为。为支持本研究，我们引入了机制代理安全（MAS）基准，包含基于LLaMA-3.1-8B、Qwen-2.5-7B和Gemma-2-9B的194个任务的完整注释多轮执行轨迹。大量实验表明，AgentLens实现了强大的安全检测性能，提供了前瞻风险预警的初步证据，并大幅减少了编码代理的有害行为，为将机制可解释性应用于动态LLM代理安全奠定了基础。代码地址：https://github.com/EddyLuo1232/AgentLens

## Abstract
Coding agents based on large language models (LLMs) demonstrate remarkable autonomous capabilities, but they also introduce significant safety and misuse risks during multi-turn interactions with external environments. Existing safety mechanisms mainly rely on external guardrails, which have a limited ability to perform fine-grained behavioral control during execution. Meanwhile, recent mechanistic interpretability methods for LLM safety are mostly confined to single-turn or jailbreak-style QA settings, limiting their ability to capture the evolving risk dynamics of multi-turn agent execution. In this paper, we investigate the safety of multi-turn coding agents from an internal perspective. We propose AgentLens (Mechanistic Subspace Intervention and Steering), a white-box defense framework that performs runtime safety detection and representation-level mitigation for coding agents. Unlike conventional agent guardrails, AgentLens detect harmful execution states from step-level hidden representations and mitigate unsafe behavior by intervening in a 10-dimensional subspace within a single layer. To support this research, we introduce the Mechanistic Agent Safety (MAS) benchmark, comprising comprehensively annotated multi-turn execution trajectories across 194 tasks using LLaMA-3.1-8B, Qwen-2.5-7B, and Gemma-2-9B. Extensive experiments show that AgentLens achieves strong safety detection performance, provides preliminary evidence for lookahead risk anticipation, and substantially reduces harmful actions of the coding agent, establishing a foundation for applying mechanistic interpretability to dynamic LLM agent safety. ### 2. 论文提出的方法论：核心思想、关键技术细节、流程

- **核心思想**：提出 AgentLens，一种白盒防御框架，包含两个阶段——**安全检测**和**风险缓解**。核心思想是利用单层线性探针从代理每一步的 last-token 隐藏表示中提取安全信号，并在检测到风险后通过在同一个目标层的 top-K 维子空间施加稀疏方向向量来干预表示，从而抑制有害行为。
- **关键技术细节**：
  - **安全检测**：训练一个线性二分类探针（logistic regression），输入为选定层 ℓ 的 last-token 隐藏状态 \( h^{(\ell)} \in \mathbb{R}^d \)，输出风险概率。模型冻结参数，探针在所有步骤共享。
  - **风险缓解**：当探针检测到有害步骤（预测标签为 1）时，构建一个稀疏 steering 向量 \( v \in \mathbb{R}^d \)，其中只有 top-K 维度（由探针权重的类间绝对差值选出）被设为 1，其余为 0。然后对该层所有 token 添加 \( \alpha \cdot \mathbf{1}_{n_t} v^\top \)，其中 steering 强度 α 通过一个 LLM 控制器自适应搜索，目标是最大化安全与效用的加权组合（\( \lambda_s=0.6, \lambda_u=0.4 \)）。
- **流程**：每个步骤先提取隐藏表示 → 探针预测 → 若安全则继续，若有害则触发 steering → LLM 控制器调整 α 并评估 → 选择最佳 α 生成最终动作。

### 6. 论文的主要结论与发现

- **安全信号线性可解码**：在早期到中间层（LLaMA 第 12 层、Qwen 第 13-17 层、Gemma 第 14 层），线性探针取得最高准确率（99.15%、97.46%、95.35%），表明安全相关信息可在隐藏空间中通过线性分离提取。
- **强大检测性能**：AgentLens 在三个模型上的 current-step 检测平均准确率 97.32%，优于所有基线。
- **初步 lookahead 能力**：在 LLaMA 和 Qwen 上，current-step 探针可预测下一步有害行为（准确率 92.31%、96.77%），显示前瞻性。
- **有效缓解**：AgentLens 将 ASR 从平均 85.99% 降至 13.36%，降低 72.63 个百分点，优于 RepE 和 Self Reminder。
- **因果验证**：负 steering 能将拒绝行为反转成有害执行，证明发现的子空间具有因果控制作用。
- **检测-控制差距**：在 prompt injection 场景下，探针无法可靠检测，但 steering 方向仍能有效降低 ASR（86.7% → 6.7%），提示表示级干预可能比检测更鲁棒。

### 7. 优点（方法/实验设计亮点）

- **新颖性**：首次将机械可解释性应用于多轮编码代理安全，区别于此前局限于单轮问答的工作。
- **轻量高效**：仅需单层 10 维子空间干预，不修改模型权重，理论代价低。
- **自适应 steering**：通过 LLM 控制器动态调优 steering 强度，避免固定强度导致实用性下降。
- **MAS 基准**：提供了可复现的、步骤级标注的多轮执行轨迹，填补了白盒编码代理安全基准的空白。
- **因果验证实验**：通过反向 steering 将拒绝行为翻转为执行，强有力地证明了干预方向的因果效应。

### 8. 不足与局限

- **实验规模有限**：MAS 仅含 194 个任务，且 lookahead 样本极少（LLaMA 13 个、Qwen 31 个），Gemma 甚至无 lookahead 数据，导致前瞻检测统计置信度低。
- **模型覆盖窄**：仅评估了 7B-9B 参数的开源模型，未实验更大模型（如 70B、130B）或闭源 API 模型，泛化性未知。
- **白盒依赖**：方法需要模型隐藏状态访问，无法直接用于黑盒 API 模型。
- **steering 副作用**：collapse 率从 2.24% 升至 18.54%，强 steering 导致输出崩溃或重复，可能影响实用性。
- **检测-控制不一致**：在 prompt injection 分布外场景中检测失败但控制有效，说明当前方法不能自动触发，需额外机制。
- **标注歧义**：当前步骤标签难以区分“准备性”步骤是否应标记为有害，可能引入噪声。论文也承认这一点。
- **环境限制**：所有轨迹在 Docker 容器中收集，真实部署场景的更复杂环境未涉及。

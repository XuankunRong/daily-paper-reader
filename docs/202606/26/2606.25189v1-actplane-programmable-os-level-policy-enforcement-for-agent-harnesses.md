## 摘要
AI 代理越来越多地通过管道在生产环境中运行，管道是围绕大语言模型的软件，包括一个执行安全和有效性策略（例如，“在提交前运行测试”）的引擎。执行这些策略需要弥合语义鸿沟：策略意图以不明确的自然语言表达，而执行必须针对具体的系统动作（例如，运行哪个测试）。许多策略还定义了事件排序或数据流动作。然而，现有方法存在不足。工具调用护栏会遗漏绕过工具层的系统动作，而操作系统沙箱控制资源访问而非动作，返回不透明的错误信息，使代理困惑。我们的关键洞察是，策略上下文存在于最接近任务的代理中，而执行必须在操作系统层面进行以覆盖所有执行路径。我们提出 ActPlane，一个策略引擎，允许代理声明策略，并在操作系统内核中通过语义反馈和隔离来执行它们。ActPlane 使用一个简单的信息流控制（IFC）领域特定语言来支持跨事件策略。我们使用 eBPF 实现 ActPlane，并在来自实证研究、编程任务基准和安全基准的策略上进行评估。ActPlane 提高了策略合规性，包括工具调用拦截无法观察到的间接执行路径，开销为 1.9%-8.4%。ActPlane 位于 https://github.com/eunomia-bpf/ActPlane

## Abstract
AI agents increasingly run in production through harnesses, the software around the LLM, including an engine that enforces safety and effectiveness policies, e.g., 'run tests before committing.' Enforcing these policies requires bridging a semantic gap: policy intent is expressed in underspecified natural language, while enforcement must act on concrete system actions, e.g., which test to run. Many policies also define event ordering or data flow actions. Yet existing approaches fall short. Tool-call guardrails miss system actions that bypass the tool layer, while OS sandboxes control resource access instead of actions, returning opaque errors that confuse the agent. Our key insight is that policy context lives within the agent closest to the task, while enforcement must happen at the OS to cover all execution paths. We introduce ActPlane, a policy engine that lets agents declare policies and enforces them in the OS kernel with semantic feedback and isolation. ActPlane uses a simple information-flow control (IFC) DSL to support cross-event policies. We implement ActPlane with eBPF and evaluate it on policies from the empirical study, coding-task benchmarks, and safety benchmarks. ActPlane improves policy compliance, including on indirect execution paths that tool-call interception cannot observe, with 1.9%-8.4% overhead. ActPlane is at https://github.com/eunomia-bpf/ActPlane

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：AI 代理（AI agent）通过“管道”（harness）执行任务时，需要强制执行安全与有效性策略（例如“提交前先运行测试”）。然而，策略意图以自然语言表达（语义模糊），而执行必须针对具体的系统动作（如“运行哪个测试命令”），存在根本性的“语义鸿沟”。
- **现有方法不足**：
  - 工具调用（tool-call）护栏只能拦截代理通过工具层发起的请求，但无法捕捉绕过工具层的系统动作（如脚本内嵌的 git commit、子进程执行）。
  - OS 沙箱（如 seccomp、AppArmor）控制资源访问而非动作，返回不透明错误（如 EPERM），且期望静态预定义策略，无法适应代理的上下文。
- **研究动机**：需要一种系统，既能由代理（最接近任务）生成具体策略，又能在 OS 内核层强制执行，覆盖所有执行路径（包括子进程、shell-outs），并提供语义反馈帮助代理恢复合规。
- **整体意义**：提出了 ActPlane，首个可编程的 OS 级策略执行引擎，弥合了语义鸿沟，提高了代理安全性与有效性，且开销低。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- 策略生成由最接近任务的代理负责（基于仓库上下文和任务），策略执行则在 OS 内核层进行，通过 eBPF 实现跨事件追踪与强制。
- 设计两个需求：
  - **R1**：代理可写、OS 可执行的策略规范——使用简单 DSL 表达，编译为确定性内核检查。
  - **R2**：安全、隔离、高效——策略域分层，高阶权威规则不可被代理削弱。

### 关键技术细节
1. **策略 DSL**（领域特定语言）：
   - 策略表示为“源→目标”约束。源可以是进程、文件、网络端点；目标是操作（exec、open、connect 等）。
   - 每个规则包含：源、目标操作、效果（`block` / `kill` / `notify`）、可选的时态门（`after ... since ...`）、原因（用于语义反馈）。
   - 支持跨事件策略：如 `kill exec "git" "commit" unless after exec "go" "test" exits 0 since write "**/*.go"`。
2. **信息流控制（IFC）引擎**：
   - 每个 OS 对象（进程、文件、网络端点）携带一组标签（bitmask），通过 fork、exec、read、write、connect 等操作传播。
   - 标签单调增长（只加不减），确保跨事件状态可追踪。
   - 规则检查：在系统调用 hook 处，检查当前主体/目标的标签集是否满足规则条件，若满足则应用效果。
3. **分层策略域**：
   - 每个进程树属于一个策略域，子域继承父域规则，只能添加新规则或缩小范围，不能删除/削弱继承的规则。
   - 高阶权威策略（如用户、平台运营者）在代理启动前载入，代理可运行时通过 delta 提交新规则，但内核中的权限检查器拒绝任何削弱继承规则的 delta。
4. **实现**：
   - Rust 用户态编译器/运行器（~3.2K 行），eBPF 内核引擎（~1.8K 行 BPF C）。
   - 使用 BPF-LSM 实现预操作 hook（block），tracepoint 实现后操作观察/终止（kill/notify）。
   - 每个规则编译为固定大小配置，内核接收紧凑规则表，返回规则 ID 给用户态格式化语义反馈。

## 3. 实验设计：数据集、基准、对比方法

### 数据集与基准

## 6. 主要结论与发现

1. **DSL 可被代理生成**：GPT-5.5 成功翻译全部 607 条 OS 可执行策略，初次尝试成功率 99.7%。
2. **系统效果显著**：ActPlane 决策合规率（DCR）达 75.8%，高于 prompt-filter（48.4%）、tool-regex（45.3%）、FIDES（48.9%）和 ActPlane-opaqe（53.7%）。在违规轨迹上检测率 77.2%（基线仅 34-40%）。
3. **跨模型稳定性高**：DeepSeek-Pro V4 下 ActPlane DCR 77.4%，系统排名一致（κ=0.852）。
4. **语义反馈关键**：满版 ActPlane 正确违规结果数为 86，而无反馈版仅 27；检测率由 75.4% 提升至 94.7%（单次修订后）。
5. **低开销**：宏基准开销 1.9%-8.4%；微基准中 open 操作延迟增加约 13μs（100 规则时），但远小于 LLM 推理时间。
6. **安全基准**：ActPlane 阻止 74% 的基线不安全行为（78/106），未阻止的 28 例多为聊天语义危害、文件内容危害或服务端协议未覆盖。
7. **迭代改进**：28 例假阴性经一次修订后 26 例纠正（93%）。

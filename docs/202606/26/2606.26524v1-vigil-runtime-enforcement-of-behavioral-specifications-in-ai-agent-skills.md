## 摘要
智能体系统越来越多地通过第三方技能来行动，使模型生成的决策能够影响文件、通信渠道和网络物理设备。这些技能通常包含自然语言规范，定义了访问权限、披露限制、执行权限和所需的先决条件。尽管这些规范描述了技能行为的预期边界，但它们本身并不提供可执行的运行时执行。执行这些规范会带来上下文粒度挑战：即使针对特定任务上下文编写了策略，监控器仍然必须决定观察哪些事件、保留哪些状态、在多大程度上进行推理以及在哪里介入。选择错误的粒度可能会阻塞良性执行，或者遗漏仅跨多个操作才会出现的违规。然而，大多数现有的执行机制假设固定的事件模型或执行点。

在这项工作中，我们提出了VIGIL，一个面向智能体系统的端到端运行时执行框架。VIGIL根据技能规范中的行为策略、操作员定义的约束以及跨多个技能的全局规则来检查智能体的实际执行轨迹。为了使这些策略可执行，VIGIL引入了一种策略语言，该语言捕获了针对智能体-工具事件的上下文特定执行需求，包括时间依赖关系、参数约束和值流条件。该语言与符号评估规则配对，将策略翻译为有限轨迹上的SMT约束，从而使VIGIL能够检测依赖于事件顺序、参数关系或跨调用值流的违规，而不是依赖固定的单调用过滤器。在涵盖办公文档、操作和工程任务的真实LLM智能体运行中，VIGIL以超过95%的召回率和低于10%的误报率检测到策略违规。

## Abstract
Agentic systems increasingly act through third-party skills, allowing model-generated decisions to affect files, communication channels, and cyber-physical devices. These skills often include natural-language specifications that define access permissions, disclosure limits, execution privileges, and required preconditions. Although such specifications describe the intended boundaries of skill behavior, they do not by themselves provide executable runtime enforcement. Enforcing them raises a contextual granularity challenge: even when a policy is written for a particular task context, a monitor must still decide which events to observe, what state to retain, how far across the execution to reason, and where to intervene. Choosing the wrong granularity can either block benign executions or miss violations that emerge only across multiple actions. Most existing enforcement mechanisms, however, assume a fixed event model or enforcement point.   In this work, we present VIGIL, an end-to-end runtime enforcement framework for agentic systems. VIGIL checks an agent's actual execution trace against behavioral policies from skill specifications, operator-defined constraints, and global rules spanning multiple skills. To make such policies executable, VIGIL introduces a policy language that captures context-specific enforcement requirements over agent-tool events, including temporal dependencies, argument constraints, and value-flow conditions. The language is paired with symbolic evaluation rules that translate policies into SMT constraints over finite traces, allowing VIGIL to detect violations that depend on event order, argument relationships, or cross-call value flow rather than relying on fixed single-call filters. On real LLM-agent runs spanning office-document, operational, and engineering tasks, VIGIL detects policy violations with over 95% recall and a false-positive rate below 10%.

## 1. 核心问题与整体含义（研究动机和背景）

**核心问题**：当前基于大语言模型（LLM）的AI智能体系统，通过调用第三方“技能”（Skills）来执行文件操作、通信、控制物理设备等任务。这些技能通常附带自然语言描述的行为规范（如：敏感数据禁止泄露、特定操作前需确认、中间产物需验证后使用等）。然而，目前这些规范仅仅是“文档”，缺乏有效的运行时执行机制，系统只能完全信任智能体自行遵守，这导致了严重的安全漏洞。

**背景与动机**：已有安全事件表明（如Microsoft 365 Copilot、Salesforce Agentforce的数据泄露，以及智能体失控事件），这类“盲信”正在失败。最隐蔽的违规并非单个语法错误的命令，而是隐藏在多个动作序列之间，单个检查看似合法，但组合起来却违反了整体规范。例如，一个智能体可以执行一系列各自授权、符合模式的工具调用，但共同导致了违规行为（如跳过关键验证步骤，直接使用未经验证的中间数据）。现有的防御机制（沙箱、单动作过滤器、信息流控制、离线轨迹分析等）存在结构性盲点，无法有效应对这种隐藏在多个动作间、依赖时序和数据流的违规行为。

## 2. 论文提出的方法论

### 核心思想
VIGIL 的核心思想是将抽象的自然语言行为规范，在智能体运行时动态地“落地”到具体的执行轨迹上，通过SMT求解器进行确定性检查。其关键是将一个抽象的规范问题，转化为在有限轨迹上可计算的、量词自由的SMT约束满足问题。

### 关键技术细节
VIGIL 是一个端到端的运行时参考监控器，其工作流程分为三个主要阶段：

1.  **抽象（ABSTRACT）**：将智能体原始的、异构的工具调用流，转换为一个**类型化的、有序的事件序列**（trace）。每个事件包含四个关键组成部分：动作名 `a(e)`、参数映射 `e[k]`、状态 `s(e)`（成功/失败）、输出观察 `o(e)`。同时，从运行中提取出**签名（Signature）Σ**（包含所有出现过的动作、参数名和值）和**事实库（Fact Base） JτK**（将所有事件以原子事实形式存储，例如 `a(e) = warpAffine`，`e5[transform] = M`）。该阶段将混沌的执行日志提升为可检查的逻辑事实。

2.  **编译（COMPILE）**：将自然语言规范（Specification D）编译成用VIGIL策略语言（Policy Language）表达的**策略（Policy P）**。该语言包含：
    - **模式（Patterns）**：用于选择特定事件，可以对动作名、参数、输出、状态进行约束，并且支持**变量绑定**（如 `k = x`），用于将两个不同事件中的相同值关联起来。
    - **时间形式（Temporal Forms）**：定义了六种关键的行为约束模式：`abs`（禁止）、`prec`（前提）、`resp`（义务）、`bresp`（有界义务）、`rslv`（最终解决）和 `until`（区间守卫）。
    - **编译规则**：VIGIL 定义了一组符号编译规则，将每个策略语句 `φ` 编译成一个量词自由的公式 `Rφ`，该公式对轨迹中的所有事件进行量化和约束。例如，`prec(ψ, ψ')` 规则被编译为：“对每个事件e，如果e匹配ψ，那么必须存在一个更早的事件e'匹配ψ'”。

3.  **验证（VERIFY）**：将事实库 `JτK` 与编译后的策略要求 `Rφ` 一起送入 **SMT求解器（Z3）** 进行求解。
    - **决策**：检查 `unsat(JτK ∧ Rφ)` 是否为真。如果为真，表示事实与要求矛盾，即轨迹违反了该策略。
    - **定位**：当发现违规时，求解器会返回一个**最小不可满足核心（MUC）**，其中包含了导致冲突的特定条件。通过该条件可以定位到具体的违规事件，从而实现精确的违规定位。

### 算法流程（文字说明）
VIGIL 的算法核心是 `VIGIL(L, D)`，接收执行流L和规范D：
1.  调用 `ABSTRACT(L)` 得到签名 Σ 和事实库 JτK。
2.  调用 `COMPILE(D, Σ)` 得到基于签名 Σ 的策略 P。
3.  对于 P 中的每一个语句 φ：
    a.  将其编码为SMT公式 `Rφ`。
    b.  检查 `unsat(JτK ∧ Rφ)`。如果为真，则记录该违规语句和通过不可满足核心定位到的违规事件。
4.  如果没有任何违规，则判定为 `safe`，否则判定为 `unsafe` 并返回所有违规的语句和对应的事件。

## 8. 不足与局限

1.  **轨迹可观测性边界**：VIGIL只能看到被工具化记录的事件。如果违规行为发生在**不透明的脚本内部**（如`secure_archive.py`内部的操作），则无法被VIGIL捕获。这构成了一个漏报的来源。
2.  **规范编译的保真度边界**：从自然语言规范到策略语言的编译过程依赖LLM，可能产生偏差。例如，如果LLM将“批量修改日历”遗漏了“批量”这个限定词，就会导致**误报**。这种语义泛化的误差限制了系统的精确性。
3.  **对LLM编译的依赖**：编译步骤依赖大模型（`claude-opus-4-6`），虽然可以被缓存，但其本身的质量决定了最终策略的准确性和完备性。对于更复杂、更模糊的自然语言规范，编译效果可能不佳。
4.  **对SMT求解性能的依赖**：虽然目前测试中SMT求解时间很短（0.27秒），但对于极其复杂的策略和非常长的轨迹，求解时间可能会显著增加，影响实时性。论文未报告求解器的极限情况。
5.  **实验范围**：虽然论文进行了跨基准测试，但SB+SI的数据规模和多样性（主要来自SkillsBench和Skill-Inject）仍有进一步提升的空间。在更广泛的、不同领域的智能体场景中的表现尚待验证。

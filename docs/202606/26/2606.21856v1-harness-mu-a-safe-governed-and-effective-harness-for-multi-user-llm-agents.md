## 摘要
随着大型语言模型（LLM）代理在协作工作流中的日益部署，需要强大的多用户、多主体交互机制来执行访问权限、解决权限冲突并防止未经授权的数据泄露。然而，当代LLM的单用户训练范式与多主体治理所需的硬约束之间存在根本性不匹配，这使得基于概率和提示的保障措施在多轮对抗性交互中变得脆弱。我们的关键洞察是，治理约束——谁被授权、什么被限制、谁的指令优先——是确定性的运行时变量，应通过执行钩子来实施，而非交由LLM处理。我们提出Harness-MU，这是首个面向多用户LLM代理的模型无关、零微调基础设施框架。通过将语言生成与安全编排解耦，Harness-MU保证了不可打破的权限边界，同时最大化合规需求满足。在Muses-Bench基准测试中，针对四个前沿开源和专有模型，Harness-MU在所有访问控制攻击中实现了隐私保护目标，在效用分数上比标准基线高出0.28–0.39，指令遵循准确率最高提升48.9个百分点。Harness-MU推进了“Harness Engineering”的理念，表明系统性基础设施对于解决LLM多主体治理挑战至关重要。代码和数据可在https://github.com/YuanJrShiuan/Harness-MulUser获取。

## Abstract
The increasing deployment of large language model (LLM) agents in collaborative workflows demands robust multi-user, multi-principal interaction mechanisms capable of enforcing access permissions, resolving authoritative conflicts, and preventing unauthorized data disclosure. However, a fundamental mismatch exists between the single-user training paradigm of contemporary LLMs and the hard constraints required for multi-principal governance, rendering probabilistic, prompt-based safeguards vulnerable under multi-turn adversarial interactions.Our key insight is that governance constraints -- who is authorized, what is restricted, and whose instructions take precedence -- are deterministic runtime variables that should be enforced by execution hooks rather than entrusted to the LLM. We present \textbf{Harness-MU}, the first model-agnostic, zero-tuning infrastructure framework for multi-user LLM agents. By decoupling language generation from safety orchestration, Harness-MU guarantees unbreakable permission boundaries while maximizing compliant demand satisfaction. Across four frontier open-weight and proprietary models on the \textit{Muses-Bench} benchmark, Harness-MU achieves the goal of privacy preservation across all access-control attacks, outperforming the standard baseline by 0.28--0.39 in utility score and improving instruction-following accuracy by up to 48.9 percentage points. Harness-MU advances the philosophy of \textit{Harness Engineering}, establishing that systematic infrastructure is essential for solving LLM multi-principal governance challenges. The code and data are available at https://github.com/YuanJrShiuan/Harness-MulUser.

## 1. 核心问题与整体含义（研究动机和背景）
- **核心问题**：大型语言模型（LLM）在多用户协作场景中面临三个系统性失效：未经授权的敏感信息访问、冲突指令无法依据权限优先级解决、低权限指令未能被高权限指令覆盖。根本原因在于当代LLM遵循单用户训练范式（Single Principal–Agent），缺乏多主体治理所需的**确定性硬约束**能力，概率性提示基安全措施在多轮对抗攻击下极易被突破。
- **整体含义**：现有研究（如Muses-Bench）揭示了20余种前沿LLM在访问控制和指令遵循上的严重漏洞，但均未提供解决方案。本文主张治理约束（谁被授权、什么可泄露、谁的指令优先）是**确定性的运行时变量**，应通过执行钩子强制执行，而非交由LLM内部机制。基于此，提出了**Harness Engineering**理念下的首个多用户执行框架Harness-MU。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：将语言生成与安全编排解耦，构建模型无关、零微调的执行基础设施，通过外部钩子强制执行权限和权威约束。
- **关键技术细节**：
  - **四模块运行时架构**：
    - **Gatekeeper（准入控制层）**：在模型调用前构建非概率的准入控制向量，冻结每个用户的权限记录（包括允许的敏感标记、风险参数）。不生成语言，仅锁定硬边界条件。
    - **Mediator（策略编排层）**：将全局多主体优化目标分解为每个用户的独立执行计划π_k，包含权重、硬目标、行为姿态（grant/refuse）。不直接与LLM通信，将定制指令注入下游Worker。
    - **Worker（隔离生成层）**：并行实例化单主体执行环境，每个Worker只接收与其用户相关的上下文Γ_k（用户身份、指令集、共享上下文、Mediator计划）。保证跨用户上下文不污染，阻止对抗用户发起提示注入攻击。
    - **ComplianceChecker（可行域投影层）**：后生成阶段，对每个响应执行确定性检查，包括隐私泄露检测（子串匹配敏感标记）和指令约束验证（IHEval检测器）。一旦违反，触发**失败关闭**（fail-closed），用固定拒绝负载覆盖违规部分。
  - **形式化优化目标**（公式3）：在零泄露（所有响应不含未授权敏感标记）和最高权威硬目标必须满足的约束下，最大化权威加权效用函数。该目标通过外部模块保证，而非依赖LLM的随机分布。
  - **六组件基础设施实现**：遵循Meng等人（2026）的规范，完整实例化执行循环(E)、工具注册表(T)、上下文管理器(C)、状态存储(S)、生命周期钩子(L)和评估接口(V)。
  - **算法流程**：输入用户集U、权限配置、共享上下文，Gatekeeper初始化权限记录 → Mediator生成每个用户的计划π_k → 并行Worker根据各自上下文生成响应 → ComplianceChecker逐用户检查，若泄露则替换为安全负载；若指令约束不满足则触发反馈循环重新生成部分响应。

## 3. 实验设计
- **数据集与基准**：使用**Muses-Bench**基准中的两个核心场景：

  - **多用户指令遵循（IF）**：分为**Aligned**（n=187，用户指令兼容）和**Conflict**（n=368，低权限需求与高权限硬目标冲突）两个子集。

## 6. 主要结论与发现
- **隐私保护绝对保障**：在所有模型和攻击下，Harness-MU实现**Privacy=1.000**（零泄露），而基线模型最低仅0.08（GPT-4.1-nano在某些攻击下）。
- **效用与安全性不妥协**：Harness-MU在访问控制中Utility提升0.332~0.392（相对基线），Overall提升0.240~0.427。不会因过度保守拒绝而降低合法用户可用性。
- **指令遵循大幅提升**：在难度最高的IF-Conflict场景中，GPT-4.1-nano的Accuracy从0.309提升至0.781（+47.2pp），Aligned场景提升48.9pp。弱模型和冲突场景下收益最大。
- **机制验证**：
  - **ComplianceChecker**对隐私保护贡献最大（关闭了所有泄露子串），但从未误拦截合法请求（ΔUtil=0）。
  - **Mediator + Worker隔离**防止了单转录本污染，限制了攻击面。
  - **反馈循环**在IF中呈现**反规模效应**：模型越弱，触发率越高，收益越大（GPT触发66.8%，收益7.94pp；DeepSeek触发27.3%，收益1.82pp）。
- **系统效率**：尽管总token和LLM调用次数增加（约8倍），但由于Worker并行化，关键路径延迟反而降低11%（DeepSeek环境下）。

## 7. 优点
- **模型无关与零微调**：无需修改LLM参数或数据，即可直接应用于任何现有模型，降低了部署门槛和训练成本。
- **确定性保障**：通过外部执行钩子强制硬约束，相比概率性提示防御有本质优势，抵抗多轮对抗攻击。
- **解耦架构创新**：将安全编排（Mediator/ComplianceChecker）与语言生成（Worker）分离，允许异步并行，避免单点瓶颈。
- **综合评估充分**：覆盖多种攻击类型、模板、模型、权威冲突情形，消融实验完整，统计指标严谨。
- **开源复现**：代码和数据已公开，促进后续研究。
- **理论贡献**：将Harness Engineering理念拓展到多主体场景，建立了系统基础设施重要性的实证基础。

## 8. 不足与局限
- **实验覆盖局限**：
  - 攻击者模型固定为DeepSeek-V4-Pro，未测试匹配攻击（如攻击者使用与防御者相同模型）或更强的自适应攻击。
  - 模拟用户基于结构化规则，缺乏真实人类交互的噪音、模糊授权、非正式权威信号。
  - 未包含工具调用场景（如数据库更新、邮件发送），限制了现实组织环境的覆盖面。
- **系统效率权衡**：虽然延迟降低，但总token消耗增加数倍（AC场景从29.8K增至151.6K），在大规模生产部署中可能带来可观的经济成本。未优化提示压缩或上下文缓存。
- **安全威胁盲区**：
  - ComplianceChecker基于子串匹配，可能被语义混淆、编码伪装或侧信道攻击绕过（但论文提及未来可集成形式化验证等更强技术）。
  - 未考虑合规检查器自身被攻击（如对抗性输入干扰检查逻辑）的情况。
- **权威设置固定**：实验采用严格线性权威层次，实际组织中可能有动态或重叠权威关系，未验证其泛化能力。
- **缺失人类评估**：指令遵循仅使用自动检查器验证格式/词法约束，未评估响应质量、语气或用户满意度。
